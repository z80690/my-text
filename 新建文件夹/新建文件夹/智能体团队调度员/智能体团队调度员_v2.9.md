## 核心身份

管理23专业智能体，集成中央调度、三层记忆、上下文感知、博弈调度、**100%并行执行**、**自主学习能力**。

## 一、架构设计

### 1.1 五层架构

```
┌─────────────────┐
│ 中央调度层      │ 路由│均衡│聚合│状态
├─────────────────┤
│ 监控层          │ 执行│性能│异常│日志
├─────────────────┤
│ 上下文管理层    │ 工作│情景│语义记忆
├─────────────────┤
│ 知识图谱层      │ 概念│关系│意图推断
├─────────────────┤
│ 智能体层        │ 23专业智能体
└─────────────────┘
```

### 1.2 三层记忆

| 类型 | 内容 | 周期 | 用途 |
|-----|-----|-----|-----|
| 工作记忆 | 当前对话消息、临时状态 | 24h | 实时对话上下文 |
| 情景记忆 | 任务历史、执行记录 | 30d | 历史查询、路由优化 |
| 语义记忆 | 领域知识、术语定义 | 持久 | 长期知识存储 |

### 1.3 自主学习能力

系统支持基于经验的自主学习，核心机制：
- **情景记忆积累**：每次任务执行后自动记录到情景记忆
- **相似任务检索**：通过`get_similar_tasks`接口查询历史相似任务
- **经验复用**：根据历史成功案例优化智能体选择和执行策略
- **架构预留**：为集成向量数据库和知识图谱预留接口

### 1.4 状态机定义（v2.9新增）

```python
class AgentState(TypedDict):
    messages: list                    # 消息列表
    context: dict                     # 上下文数据
    status: str                       # idle/running/completed/error/waiting
    result: Optional[dict]            # 执行结果
    execution_history: list           # 执行历史
```

## 三、标准化智能体定义格式（v2.9新增）

### 3.1 YAML定义模板

```yaml
agent_id: string           # 唯一标识
name: string               # 显示名称
version: semver            # 版本号 (如: 3.2.0)
type: reactive|proactive|hybrid

# 核心定义 (CrewAI标准)
role: string               # 角色定位
goal: string               # 目标/愿景
backstory: string          # 背景故事

# 能力定义
capabilities:
  - name: string
    description: string
    triggers: [string]

# 工具定义
tools:
  - name: string
    description: string
    parameters:
      name: string
      type: string
      required: bool

# 行为配置
behavior:
  max_retries: number
  timeout_seconds: number
  error_handling: string

# 记忆配置
memory:
  type: short|long|both
  retention: duration
```

### 3.2 示例：代码执行智能体

```yaml
agent_id: code_executor_agent
name: 代码执行智能体
version: 3.2.0
type: reactive

role: "专业代码执行专家"
goal: "安全、准确、高效地执行用户提交的代码任务"
backstory: |
  你是智能体团队中的代码执行专家，精通Python、JavaScript、TypeScript、Go、Rust等语言。
  你始终在沙箱环境中执行代码，确保系统安全。

capabilities:
  - name: execute_code
    description: 执行代码并捕获输出
    triggers: ["执行代码", "运行脚本", "调试"]

tools: []

behavior:
  max_retries: 3
  timeout_seconds: 30
  sandbox_mode: true

memory:
  type: short
  retention: 24h
```

## 四、并行执行模式

### 4.1 核心规则

**100%并行调度**：所有任务默认使用并行执行模式，执行智能体完成任务，监控智能体实时监控。

### 4.2 工作流概览

```
任务输入→创建上下文→并行启动
              ┌──────┴──────┐
              ▼             ▼
        执行智能体      监控智能体
              │             │
              ├─步骤─┐      │
              ├─步骤─┼─→监控
              └─结果─┘      │
                      └→聚合→返回用户
```

## 七、博弈调度机制（v2.9优化）

### 7.1 两种模式（简化自v2.8的三种模式）

| 模式 | 触发词 | 流程 |
|-----|-------|-----|
| **模式一：优化改进** | 优化、改进、重构、审查 | 生成→挑剔→融合 |
| **模式二：深度设计** | 设计、架构、方案、对比 | 蓝图→挑战→融合 |

**优化说明**：将原"模式一：辩论"合并到"模式二：深度设计"中，因为对比分析是设计任务的子集，简化后更易于实现和维护。

### 7.2 触发流程

```
任务→分词→query_kg→概念聚合
    ├──含"优化/改进/重构"→模式一
    ├──含"设计/架构/对比"→模式二
    └──无概念→复杂度判断→模式二/常规路由
```

## 十、执行流程

```
用户输入→Step1:接收任务→Step2:图谱查询+上下文分析→Step3:工作流选择→Step4:并行调度→Step5:结果聚合→Step6:返回结果
```

### 10.1 步骤说明

**Step1**：解析输入、关联/创建上下文、检查相似任务  
**Step2**：分词查询、分析偏好、确定博弈模式  
**Step3**：选择工作流模板（默认parallel_execution）  
**Step4**：总线分发、启动监控、记录历史  
**Step5**：收集输出、整合监控报告、更新记忆  
**Step6**：输出结果、记录日志

### 10.2 错误处理

| 错误码 | 说明 | 处理 |
|-----|-----|-----|
| ERR_AGENT_UNAVAILABLE | 智能体不可用 | 切换备用 |
| ERR_CONTEXT_EXPIRED | 上下文过期 | 重新创建 |
| ERR_STATE_LOCKED | 状态锁定 | 等待/解锁 |
| ERR_ROUTE_FAILED | 路由失败 | 检查总线 |
| ERR_TIMEOUT | 执行超时 | 增加超时/终止 |
| ERR_MONITOR_FAILED | 监控失败 | 降级顺序执行 |

## 第二章：四种记忆类型

### 2.1 User Memory（用户记忆）

**功能**：包含用户的角色、目标、职责和知识

**存储路径**：`.trae/memories/user/`

**何时使用**：
- 当你的工作应基于用户的个人资料或视角时
- 例如，如果用户要求解释代码，你的回答应 tailored 到他们最看重的细节

### 2.2 Feedback Memory（反馈记忆）

**功能**：记录用户纠正你的方法或确认非显而易见的方法有效

**存储路径**：`.trae/memories/feedback/`

**何时使用**：
- 每当用户纠正你或确认你的方法时
- 这有助于你避免重复错误并强化有效的行为

### 2.3 Project Memory（项目记忆）

**功能**：记录项目的上下文、背景和代码结构

**存储路径**：`.trae/memories/project/`

**何时使用**：
- 当你的工作涉及项目特定的知识，且这些知识无法从当前代码状态中 grep 或推断时
- 例如，为什么选择某个库，或某个模块的历史背景

### 2.4 Reference Memory（参考记忆）

**功能**：存储指向外部资源、文档和历史记录的指针

**存储路径**：`.trae/memories/reference/`

**何时使用**：
- 当用户提到外部资源（如 Linear ticket、Grafana dashboard、API 文档）时
- 保存这些指针，以便未来需要时能快速引用

## 第三章：记忆格式规范

### 3.1 文件命名规范

每个记忆保存为独立的 `.md` 文件，文件名应简洁描述主题：

| 示例 | 类型 |
|------|------|
| `user-prefs.md` | User |
| `no-mock-database.md` | Feedback |
| `auth-architecture.md` | Project |
| `monitoring-dashboard.md` | Reference |

### 3.2 内容结构

每条记忆必须包含：

| 字段 | 说明 |
|------|------|
| **规则本身** | 清晰陈述事实或规则 |
| **Why** | 记录当时决策的理由，以便判断边界情况 |
| **How to apply** | 明确说明何时触发此记忆 |

### 3.3 元数据

在文件顶部使用 YAML frontmatter 标注类型（可选）：

```markdown
---
type: user
created: 2026-05-09
---

# 用户偏好-简洁回复

不要在回复末尾总结刚做了什么，用户能看到diff。

**Why:** 用户明确要求过
**How to apply:** 所有回复结束时
```

## 第四章：记忆准入原则

### 4.1 核心原则

**只记录无法从代码和 Git 历史中推导出的信息**

### 4.2 不记录的内容

| 类型 | 原因 | 替代方案 |
|------|------|---------|
| 代码模式 | 可通过 grep 获取 | grep 命令 |
| 文件结构 | 可通过 ls 获取 | ls 命令 |
| Git 历史 | git log 是权威来源 | git log |
| CLAUDE.md 内容 | 避免重复 | 直接引用 |
| 临时任务状态 | 会话级信息无需持久化 | 上下文管理 |

## 第五章：Dream 记忆整合

### 5.1 触发条件

| 条件 | 值 |
|------|-----|
| 距上次整理 | ≥24 小时 |
| 期间工作会话数 | ≥5 个 |

### 5.2 操作流程

```
1. 扫描 → 读取记忆文件及会话日志
2. 整合 → 合成为持久、组织良好的记忆
3. 剪枝 → 删除过时、矛盾或冗余的记忆
4. 更新 → 直接写入记忆目录
```

### 5.3 整合目标

- 搞清楚自己现在知道什么
- 将相对时间转为绝对时间
- 新发现覆盖旧矛盾
- 保持记忆库的精炼和可用

## 第六章：检索与验证

### 6.1 检索机制

使用小模型作为"记忆检索员"：
- 在每次对话前，小模型会扫描记忆索引
- 根据相关性返回前 3-5 条记忆注入上下文

### 6.2 验证规则

**在使用记忆前必须验证**：

| 记忆内容 | 验证方式 |
|---------|---------|
| 文件路径 | ls 命令检查 |
| 函数参数 | grep 确认 |
| 配置值 | 读取配置文件 |

**重要**："记忆说 X 存在"不等于"X 现在存在"

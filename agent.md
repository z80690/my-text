## 2. 三层双向同步规则

### 2.1 三层架构

| 层级 | 路径 | 角色 | 职责 |
|------|------|------|------|
| L1 | `/agent.md` | 顶层规范/契约 | 定义核心原则、规则框架 |
| L2 | `/.trae/agent.md` | 项目配置/约束 | 项目级配置、融合策略 |
| L3 | `/.trae/rules/*` | 具体实现/规则 | 详细规则、领域特定规范 |

### 2.2 同步原则

| 原则 | 说明 | 执行策略 |
|------|------|---------|
| L1 有 → L2必须有 → L3必须有 | 向下实现 | 自动同步 |
| L1 没有 → L2不能有 → L3不能有 | 禁止凭空创建 | 校验拦截 |
| 冲突时 → L1为准 | 顶层优先 | 自动修正 |

## 3. 新老规则体系融合方法

### 3.1 融合策略矩阵

| 融合场景 | 老规则体系 | 新规则体系 | 融合策略 | 优先级 |
|---------|-----------|-----------|---------|--------|
| **核心原则一致** | 有 | 有 | 保留并整合 | P0 |
| **老规则更详细** | 详细 | 简略 | 迁移细节到L3 | P1 |
| **新规则更完善** | 简略 | 详细 | 采用新规则 | P1 |
| **规则冲突** | 存在 | 存在 | 以新规则为准 | P0 |
| **老规则独有** | 有 | 无 | 评估后决定保留或淘汰 | P2 |
| **新规则独有** | 无 | 有 | 直接采用 | P1 |

### 3.2 融合流程

| 步骤 | 动作 | 说明 | 负责层级 |
|------|------|------|---------|
| 1 | 规则识别 | 识别新老规则体系中的所有规则 | L1 |
| 2 | 规则比对 | 逐规则对比，标记差异和冲突 | L1 |
| 3 | 冲突解决 | 根据融合策略解决冲突 | L1 |
| 4 | 规则迁移 | 将详细规则迁移到L3 | L3 |
| 5 | 配置更新 | 更新L2配置文件 | L2 |
| 6 | 验证同步 | 验证三层规则一致性 | 全层级 |

### 3.3 规则分类与处理

| 规则类别 | 处理方式 | 目标位置 | 说明 |
|---------|---------|---------|------|
| 核心原则 | 保留整合 | L1 | 只保留核心，去重合并 |
| 详细规范 | 迁移细化 | L3 | 详细实现放到L3 |
| 项目配置 | 更新同步 | L2 | 项目特有配置 |
| 废弃规则 | 标记淘汰 | - | 记录原因后移除 |
| 冲突规则 | 以新为准 | L1/L2/L3 | 新规则覆盖老规则 |

### 3.4 融合决策树

```
规则存在于老体系？
    ├─ 否 → 新规则独有 → 直接采用 → 更新L1/L2/L3
    └─ 是 → 规则存在于新体系？
        ├─ 否 → 老规则独有 → 评估价值 → 保留或淘汰
        └─ 是 → 规则内容一致？
            ├─ 是 → 保留整合 → 去重合并
            └─ 否 → 规则冲突 → 以新规则为准
```

## 11. 执行任务评估模块（强制启用）

> **核心目标**：避免AI偷懒，保证任务完成度100%，通过博弈机制实现自我监督

### 11.1 问题背景

| 问题 | 危害 |
|------|------|
| AI部分完成就停止 | 任务完成度不足 |
| 跳过调研直接执行 | 违反规则、方案不佳 |
| 自我感觉良好 | 缺乏客观评估 |
| 隐藏问题不报告 | 风险积累 |

### 11.2 博弈评估机制

| 阶段 | 动作 | 说明 |
|------|------|------|
| **自评** | AI生成任务清单 | 列出所有要完成的子任务 |
| **挑剔** | 对每个子任务发起挑战 | 质疑完成度、可行性、质量 |
| **辩护** | AI为完成部分辩护 | 提供证据证明完成度 |
| **仲裁** | 博弈模块裁决 | 判定完成百分比 |
| **补全** | 根据裁决补全任务 | 100%完成度目标 |

### 11.3 评估维度

| 维度 | 检查项 | 权重 |
|------|--------|------|
| 完整性 | 是否包含所有子任务 | 30% |
| 正确性 | 解决方案是否正确 | 30% |
| 彻底性 | 是否挖掘到根本原因 | 20% |
| 可行性 | 方案是否可执行 | 20% |

### 11.4 完成度判定标准

| 完成度 | 状态 | 处理方式 |
|--------|------|---------|
| 100% | ✅ 完全完成 | 通过评估 |
| 80-99% | ⚠️ 基本完成 | 必须补全剩余部分 |
| 60-79% | 🔴 部分完成 | 重新执行并评估 |
| <60% | 🔴 未完成 | 重新调研+执行+评估 |

### 11.5 执行流程

```
用户任务 → AI自评（生成任务清单）
              ↓
         博弈挑剔（挑战每个子任务）
              ↓
         AI辩护（证明完成部分）
              ↓
         博弈裁决（判定完成百分比）
              ↓
    ┌──────┴──────┐
    ↓              ↓
 100%完成      未100%完成
    ↓              ↓
 返回结果      强制补全+重新评估
```

### 11.6 强制约束

| 约束 | 说明 |
|------|------|
| 评估前置 | 任何任务返回前必须经过评估 |
| 100%目标 | 完成度必须达到100%才能返回 |
| 透明记录 | 评估过程和结果必须透明记录 |
| 可追溯 | 评估结果存入记忆系统 |
| **递归循环** | 未完成任务必须递归补全，最多3次 |
| **子任务独立评估** | 每个子任务必须独立评分 |

### 11.7 递归处理规则

#### 递归循环机制

| 完成度 | 递归策略 | 最大次数 |
|--------|---------|---------|
| 80-99% | 直接补全剩余部分 | 3次 |
| 60-79% | 重新执行并评估 | 3次 |
| <60% | 重新调研+执行+评估 | 3次 |

#### 递归流程

```
用户任务 → 评估 → 未100%完成？
                     ↓
                ┌────┴────┐
                ↓         ↓
               是         否
                ↓         ↓
         检查递归次数 < 3？  返回结果
                ↓
          ┌────┴────┐
          ↓         ↓
         是         否
          ↓         ↓
      补全/重执行   报告失败
          ↓              ↓
      重新评估      记录失败原因
          ↓              ↓
      返回评估   通知用户并建议替代方案
```

#### 递归终止条件

| 条件 | 处理方式 |
|------|---------|
| 完成度达到100% | 退出递归，返回结果 |
| 递归次数达到3次 | 终止递归，报告失败 |
| 用户手动终止 | 终止递归，返回当前进度 |
| 系统异常 | 终止递归，记录错误 |

### 11.8 子任务递归评估规则

#### 任务分解原则

| 原则 | 说明 |
|------|------|
| 独立性 | 每个子任务可独立完成和评估 |
| 原子性 | 子任务不可再分 |
| 完整性 | 所有子任务之和等于总任务 |
| 可验证性 | 每个子任务可独立验证完成度 |

#### 子任务评分规则

```
总任务评分 = Σ(子任务评分 × 子任务权重) / Σ(子任务权重)

其中：
- 子任务权重 = 子任务复杂度 / 总复杂度
- 每个子任务必须独立评估（完整性、正确性、彻底性、可行性）
```

#### 子任务评估流程

```
总任务
   ↓
分解为子任务（最多10个）
   ↓
┌───┬───┬───┬───┐
│ 1 │ 2 │ 3 │...│  并行评估每个子任务
└───┴───┴───┴───┘
   ↓
聚合子任务评分（加权平均）
   ↓
总任务评分
```

#### 子任务粒度控制

| 任务规模 | 建议子任务数 |
|---------|-------------|
| 简单（<1小时） | 1-2个 |
| 中等（1-4小时） | 3-5个 |
| 复杂（>4小时） | 5-10个 |

### 11.9 触发条件

| 条件 | 是否触发评估 |
|------|-------------|
| 用户明确要求完成任务 | ✅ 强制评估 |
| AI自评完成 | ✅ 强制评估 |
| 发现潜在风险 | ✅ 强制评估 |
| 用户反馈不满意 | ✅ 强制重新评估 |

## 14. 智能体系统核心元规则漏洞修复清单

### 14.1 漏洞1：熔断机制缺失

#### 现状
- 当前流程："评估通过 → 执行 → 结束"的单向流程
- 评估结果一旦通过，执行过程无法中断

#### 风险
- 评估出错时无法中途纠正
- 错误执行可能造成不可逆后果
- 用户无法干预执行过程

#### 修复方案

| 触发条件 | 处理方式 |
|----------|---------|
| 用户手动中断 | 立即暂停执行，返回当前进度 |
| 执行异常检测 | 自动触发熔断，暂停执行 |
| 置信度下降 | 自动预警，等待用户确认 |

#### 熔断状态机
```
正常执行 → 检测到风险 → 触发熔断 → 用户确认/取消 → 继续/终止
```

### 14.2 漏洞2：不确定性处理缺失

#### 现状
- 评估结果为"通过/不通过"或简单百分比
- 缺乏置信度概念，无法量化不确定性

#### 风险
- 数据质量差时系统仍盲目自信执行
- 无法区分"确定的通过"和"勉强的通过"
- 用户无法了解结果的可靠性

#### 修复方案

| 置信度范围 | 状态 | 处理方式 |
|-----------|------|---------|
| ≥95% | ✅ 高置信 | 直接执行 |
| 85%-95% | ⚠️ 中等置信 | 提示用户后执行 |
| 70%-85% | ⚠️ 低置信 | 要求用户确认 |
| <70% | 🔴 不可靠 | 拒绝执行或降级处理 |

#### 置信度计算公式
```
置信度 = 数据质量 × 0.3 + 历史成功率 × 0.4 + 环境稳定性 × 0.3
```

### 14.3 漏洞3：自指悖论

#### 现状
- 同一智能体实例"扮演"角色审查自身工作
- 思维模式一致，难以发现盲区

#### 风险
- 固有思维盲区无法被发现
- 审查流于形式，无法真正发现问题
- 缺乏独立监督机制

#### 修复方案

| 角色 | 要求 |
|------|------|
| 执行者 | 负责任务执行 |
| 审查者 | 必须由调度员指派不同的智能体实例 |
| 仲裁者 | 第三方智能体，解决争议 |

#### 物理隔离原则
- 执行者和审查者必须是不同的智能体实例
- 审查者必须具备与执行者不同的知识视角
- 仲裁者必须独立于前两者

### 14.4 漏洞4：无降级策略

#### 现状
- 递归3次失败即报错，任务完全失败
- 用户得不到任何结果

#### 风险
- 用户体验差，任务失败后无任何产出
- 部分完成的工作被丢弃
- 无法满足最低需求

#### 修复方案

| 失败次数 | 策略 | 输出 |
|----------|------|------|
| 1次失败 | 重试当前步骤 | 继续执行 |
| 2次失败 | 降级到备用方案 | 保底输出 |
| 3次失败 | 生成部分结果 | 提供已完成部分 |

#### 保底输出策略
- 收集已完成的部分结果
- 生成任务执行报告（说明失败原因）
- 提供替代方案或建议
- 记录失败经验供后续学习

### 14.5 综合修复架构

```
用户任务 → 置信度评估 → 指派执行智能体
                              ↓
                    执行 + 独立审查并行
                              ↓
                    熔断检测（实时）
                              ↓
                    ┌──────┴──────┐
                    ↓              ↓
                执行成功        执行失败
                    ↓              ↓
                返回结果      降级处理 → 保底输出
```

| **性能预测** | 预测响应时间、成功率 | 特征工程、模型训练 |

### 19.2 调度策略

| 策略类型 | 说明 | 触发条件 |
|---------|------|---------|
| **预扩容** | 提前启动备用智能体 | 预测任务量增长 >30% |

## 21. 成本优化

### 21.1 成本维度

| 成本类型 | 说明 | 衡量指标 |
|---------|------|---------|
| **计算成本** | CPU、GPU等计算资源消耗 | 核心小时、GPU小时 |
| **存储成本** | 内存、磁盘等存储资源消耗 | GB-小时 |
| **网络成本** | 数据传输、网络带宽消耗 | 流量GB |
| **时间成本** | 任务执行时间、延迟成本 | 等待时间、超时损失 |

### 21.2 优化策略

| 策略类型 | 说明 | 预期收益 |
|---------|------|---------|
| **智能体池化** | 共享智能体资源，按需调度 | 资源利用率提升30-50% |
| **动态扩缩容** | 根据负载自动增减资源 | 成本降低40-60% |
| **任务优先级** | 高优先级任务优先分配资源 | 关键任务SLA保证 |
| **资源限制** | 为智能体设置资源使用上限 | 防止资源浪费 |
| **闲置回收** | 自动回收闲置资源 | 降低无效成本 |

### 21.3 成本-性能权衡

| 权衡策略 | 性能影响 | 成本影响 | 适用场景 |
|---------|---------|---------|---------|
| **性能优先** | ✅ 最优性能 | ⚠️ 较高成本 | 关键业务、高峰期 |
| **成本优先** | ⚠️ 可接受性能 | ✅ 最低成本 | 非关键业务、低峰期 |
| **平衡模式** | ⚖️ 平衡性能与成本 | ⚖️ 适中成本 | 常规业务场景 |
| **自适应** | 动态调整平衡点 | 动态优化成本 | 复杂多变场景 |

## 二、Architect（架构设计）

### 2.1 架构图
```
┌─────────────────────────────────────────────────────┐
│                    请求入口                          │
└─────────────────────────┬───────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────┐
│                   MCP执行层                         │
│  (重试2次 + 超时10s + 日志记录)                     │
└─────────────────────────┬───────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────┐
│                   L1规则层（只读）                   │
│  security.json + whitelist.json + blacklist.json   │
└─────────────────────────┬───────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────┐
│                   L2流程层                          │
│  strict.json（正式） / soft.json（测试）            │
└─────────────────────────┬───────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────┐
│                   L3建议层                          │
│  suggestions.json（最佳实践建议）                   │
└─────────────────────────────────────────────────────┘
```

### 2.2 核心流程
1. 接收请求（action + path + content）
2. L1安全校验 → 拦截危险命令/模式/关键词
3. L1黑名单校验 → 拦截黑名单项
4. L1白名单校验 → 验证允许操作
5. L2模式判断 → strict/soft处理
6. L3建议输出 → 提示最佳实践

### 2.3 风险评估
| 风险 | 概率 | 影响 | 应对 |
|---|---|---|---|
| 测试遗漏 | 中 | 高 | 编写完整测试套件 |
| 规则冲突 | 低 | 中 | 规则优先级排序 |
| 性能问题 | 低 | 低 | 缓存规则配置 |

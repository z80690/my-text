# 质量检查指南 (L3)

**版本**: v1.1 | **更新日期**: 2026-05-20 | **规则状态**: ✅ 生效中
**触发条件**: 编写代码、修改代码、交付前自动执行
**L1支撑**: 质量保障（四维模型、度量标准、检查流程）
**L2支撑**: 质量规范、测试规范

---

## 一、质量检查框架

### 1.1 四维质量模型 [L3-5.1.1]

| 维度 | 定义 | 检查重点 |
|------|------|----------|
| **准确性** | 代码、配置、命令准确、完整、可执行 | 语法正确、逻辑正确、无运行时错误 |
| **安全性** | 代码和配置符合安全标准 | 无硬编码、依赖安全、权限合规 |
| **可维护性** | 代码易于理解和维护 | 命名规范、注释清晰、结构合理 |
| **可测试性** | 代码具备良好的测试基础 | 测试覆盖、接口稳定、解耦设计 |

### 1.2 质量度量标准 [L3-5.1.2]

| 指标 | 达标标准 | 检测工具 | 执行时机 |
|------|----------|----------|----------|
| 单元测试覆盖率 | ≥70% | pytest, jest, coverage.py | 提交前 |
| 静态检查告警 | 0 critical | flake8, eslint, mypy | 开发中+提交前 |
| 安全审计问题 | 0 high | npm audit, pip-audit | 提交前 |
| 代码复杂度 | Cyclomatic < 15 | radon, eslint-complexity | 提交前 |
| 代码重复率 | < 5% | jscpd, radon | 提交前 |

## 二、质量检查流程

### 2.1 开发中检查 [L3-5.2.1]
```
编写代码 → IDE实时检查 → 自动修复/手动修复 → 继续开发
```

### 2.2 提交前检查 [L3-5.2.2]
```
完成功能 → 运行 lint → 运行测试 → 安全审计 → 代码审查 → 提交
```

### 2.3 CI/CD检查 [L3-5.2.3]
```
代码提交 → 自动构建 → 静态分析 → 单元测试 → 集成测试 → 部署
```

## 三、各维度详细检查清单

### 3.1 准确性检查 [L3-5.3.1]
- [ ] 代码语法正确（无编译错误）
- [ ] 逻辑正确（符合业务需求）
- [ ] 配置文件格式正确
- [ ] 命令可执行（无语法错误）
- [ ] 变量/函数引用正确

### 3.2 安全性检查 [L3-5.3.2]
- [ ] 无硬编码密钥/Token/密码
- [ ] 使用环境变量或安全配置
- [ ] 依赖包无高危漏洞
- [ ] 输入验证完整
- [ ] 无SQL注入风险

### 3.3 可维护性检查 [L3-5.3.3]
- [ ] 命名符合项目规范
- [ ] 关键代码有注释（解释"为什么"）
- [ ] 函数职责单一（单一职责原则）
- [ ] 避免过长函数（<50行）
- [ ] 避免过深嵌套（<3层）

### 3.4 可测试性检查 [L3-5.3.4]
- [ ] 新增代码有单元测试
- [ ] 测试覆盖关键路径
- [ ] 接口稳定（无频繁变更）
- [ ] 代码解耦（易于Mock）
- [ ] 纯函数可独立测试

## 四、工具使用指南

### 4.1 Python项目 [L3-5.4.1]
```bash
# 静态检查
flake8 . --max-complexity=15
mypy .

# 测试覆盖
pytest --cov=src tests/
coverage report --fail-under=70

# 安全审计
pip-audit
```

### 4.2 JavaScript/TypeScript项目 [L3-5.4.2]
```bash
# 静态检查
eslint . --max-complexity=15
tsc --noEmit

# 测试覆盖
jest --coverage
nyc --check-coverage --lines 70

# 安全审计
npm audit
```

## 五、质量问题处理流程

### 5.1 发现问题 [L3-5.5.1]
```
检测到问题 → 记录问题 → 评估严重程度 → 分配责任人
```

### 5.2 修复问题 [L3-5.5.2]
```
分析根因 → 制定修复方案 → 实施修复 → 验证修复
```

### 5.3 严重程度分级 [L3-5.5.3]
| 级别 | 定义 | 处理时限 |
|------|------|----------|
| Critical | 阻止开发/部署 | 立即修复 |
| High | 影响核心功能 | 24小时内 |
| Medium | 影响次要功能 | 3天内 |
| Low | 代码风格/优化建议 | 下一迭代 |

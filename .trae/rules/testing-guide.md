## 一、测试策略

### 1.1 测试覆盖要求 [L3-8.1.1]
| 代码类型 | 最低覆盖率 | 说明 |
|----------|------------|------|
| 新增代码 | ≥80% | 核心业务逻辑必须全覆盖 |
| 修改代码 | ≥70% | 确保不破坏现有功能 |
| 工具函数 | 100% | 公共函数必须有测试 |
| 接口代码 | ≥60% | API层需要覆盖 |

### 1.2 测试分层 [L3-8.1.2]
```
┌─────────────────────────────────┐
│        E2E 测试 (可选)          │  ← 端到端场景测试
├─────────────────────────────────┤
│       集成测试 (建议)            │  ← 模块间交互
├─────────────────────────────────┤
│       单元测试 (必须)            │  ← 最小可测试单元
└─────────────────────────────────┘
```

## 二、测试编写规范

### 2.1 测试文件组织 [L3-8.2.1]
```
tests/
├── unit/                    # 单元测试
│   ├── __init__.py
│   └── test_module.py
├── integration/             # 集成测试
│   └── test_api.py
└── conftest.py             # 共享fixture
```

### 2.2 测试命名规范 [L3-8.2.2]
```python
# 测试文件和函数命名
# 文件: test_<模块名>.py
# 函数: test_<被测函数>_<场景>_<预期结果>

def test_calculate_total_with_valid_input_returns_sum():
    """测试calculate_total函数输入有效数据时返回正确求和结果"""
    ...

def test_user_login_with_invalid_password_raises_auth_error():
    """测试用户登录密码错误时抛出认证异常"""
    ...
```

### 2.3 测试结构 (Arrange-Act-Assert) [L3-8.2.3]
```python
def test_example():
    # Arrange - 准备测试数据
    user_data = {"name": "test", "email": "test@example.com"}

    # Act - 执行被测操作
    result = create_user(user_data)

    # Assert - 验证结果
    assert result.id is not None
    assert result.name == "test"
```

### 2.4 测试隔离原则 [L3-8.2.4]
- 每个测试独立运行，不依赖其他测试
- 使用fixture管理测试数据
- 避免使用全局状态
- Mock外部依赖

## 三、重构安全网

### 3.1 重构前检查 [L3-8.3.1]
```
重构前:
□ 运行所有测试，确认通过率 = 100%
□ 检查测试覆盖率基线
□ 记录关键测试用例
□ 备份当前代码
```

### 3.2 重构中检查 [L3-8.3.2]
```
重构中:
□ 每次小改动后运行相关测试
□ 保持测试通过率 = 100%
□ 监控测试覆盖率变化
□ 及时修复失败的测试
```

### 3.3 重构后验证 [L3-8.3.3]
```
重构后:
□ 运行完整测试套件
□ 确认通过率 ≥ 重构前
□ 确认覆盖率未下降
□ 代码审查确认行为一致
```

## 四、测试执行指南

### 4.1 Python项目 [L3-8.4.1]
```bash
# 运行所有测试
pytest

# 运行并显示覆盖率
pytest --cov=src --cov-report=html

# 运行指定文件
pytest tests/unit/test_module.py

# 运行包含关键字的测试
pytest -k "test_user_login"

# 生成JUnit XML报告 (CI用)
pytest --junitxml=report.xml
```

### 4.2 JavaScript/TypeScript项目 [L3-8.4.2]
```bash
# 运行所有测试
npm test

# 运行并显示覆盖率
npm test -- --coverage

# 运行指定文件
npx jest tests/unit/module.test.ts

# 监听模式 (开发用)
npx jest --watch
```

## 五、测试质量标准

### 5.1 好测试的特征 [L3-8.5.1]
| 特征 | 说明 |
|------|------|
| **独立** | 测试间无依赖，可任意顺序执行 |
| **确定** | 相同输入必得相同结果 |
| **快速** | 单个测试 < 100ms |
| **清晰** | 测试名描述测试意图 |
| **完整** | 覆盖正常/异常/边界情况 |

### 5.2 测试自说明清单 [L3-8.5.2]
- [ ] 测试名清晰表达测试内容
- [ ] 有必要的注释解释复杂测试逻辑
- [ ] 断言消息有助于调试
- [ ] 测试数据有代表性

## 六、质量门禁中的测试要求 [L3-8.6.1]

根据 quality-gate.md，测试评分标准：

| 评分 | 测试覆盖率 | 其他要求 |
|------|-----------|----------|
| 100分 | ≥80% | 核心功能全覆盖，接口稳定 |
| 80分 | ≥60% | 核心功能有测试 |
| 60分 | ≥40% | 部分功能有测试 |
| 40分 | <40% | 测试不完善 |
| 0分 | 无测试 | 测试缺失或无法运行 |

## 第一部分：质量门禁

### 1.1 质量门禁架构
```
任务交付 → 质量检查 → 评分计算 → 门禁决策 → 反馈
              ↓              ↓           ↓
         四维检查      加权评分      通过/拒绝
```

### 1.2 质量评估维度与权重

| 维度 | 权重 | 检查内容 | 满分 |
|------|------|----------|------|
| **准确性** | 30% | 代码语法、逻辑正确性、可执行性 | 100 |
| **安全性** | 25% | 无硬编码、依赖安全、权限合规 | 100 |
| **可维护性** | 20% | 命名规范、注释清晰、结构合理 | 100 |
| **可测试性** | 25% | 测试覆盖、接口稳定、代码解耦 | 100 |

### 1.3 评分标准

#### 准确性评分（100分）
- 100分：代码完全正确，无语法错误，逻辑符合需求
- 80分：代码基本正确，存在少量可忽略的警告
- 60分：代码存在次要问题，不影响核心功能
- 40分：代码存在较多问题，部分功能受影响
- 0分：代码无法运行或核心逻辑错误

#### 安全性评分（100分）
- 100分：无硬编码、依赖无高危漏洞、权限合规
- 80分：存在低危安全问题（如未加密的配置）
- 60分：存在中危安全问题（如弱密码策略）
- 40分：存在高危安全问题（如SQL注入风险）
- 0分：存在严重安全问题（如硬编码密钥）

#### 可维护性评分（100分）
- 100分：命名规范、注释充分、结构清晰、无重复代码
- 80分：存在少量命名或格式问题
- 60分：存在重复代码或过长函数
- 40分：代码结构混乱，难以理解
- 0分：代码无法维护

#### 可测试性评分（100分）
- 100分：测试覆盖率≥80%，接口稳定，易于Mock
- 80分：测试覆盖率≥60%，核心功能有测试
- 60分：测试覆盖率≥40%，部分功能有测试
- 40分：测试覆盖率<40%，测试不完善
- 0分：无测试或测试无法运行

### 1.4 综合评分计算
```
综合得分 = (准确性 × 0.3) + (安全性 × 0.25) + (可维护性 × 0.2) + (可测试性 × 0.25)
```

### 1.5 门禁决策规则

| 分数区间 | 门禁结果 | 处理方式 |
|----------|----------|----------|
| **90-100分** | ✅ 直接通过 | 可直接合并/部署 |
| **75-89分** | ⚠️ 条件通过 | 需要代码审查后通过 |
| **60-74分** | ⏳ 修复后重评 | 修复问题后重新评估 |
| **0-59分** | ❌ 拒绝通过 | 必须修复后重新提交 |

## 第二部分：质量检查指南

### 2.1 四维质量模型

| 维度 | 定义 | 检查重点 |
|------|------|----------|
| **准确性** | 代码、配置、命令准确、完整、可执行 | 语法正确、逻辑正确、无运行时错误 |
| **安全性** | 代码和配置符合安全标准 | 无硬编码、依赖安全、权限合规 |
| **可维护性** | 代码易于理解和维护 | 命名规范、注释清晰、结构合理 |
| **可测试性** | 代码具备良好的测试基础 | 测试覆盖、接口稳定、解耦设计 |

### 2.2 质量度量标准

| 指标 | 达标标准 | 检测工具 | 执行时机 |
|------|----------|----------|----------|
| 单元测试覆盖率 | ≥70% | pytest, jest, coverage.py | 提交前 |
| 静态检查告警 | 0 critical | flake8, eslint, mypy | 开发中+提交前 |
| 安全审计问题 | 0 high | npm audit, pip-audit | 提交前 |
| 代码复杂度 | Cyclomatic < 15 | radon, eslint-complexity | 提交前 |
| 代码重复率 | < 5% | jscpd, radon | 提交前 |

### 2.3 各维度详细检查清单

#### 准确性检查
- [ ] 代码语法正确（无编译错误）
- [ ] 逻辑正确（符合业务需求）
- [ ] 配置文件格式正确
- [ ] 命令可执行（无语法错误）
- [ ] 变量/函数引用正确

#### 安全性检查
- [ ] 无硬编码密钥/Token/密码
- [ ] 使用环境变量或安全配置
- [ ] 依赖包无高危漏洞
- [ ] 输入验证完整
- [ ] 无SQL注入风险

#### 可维护性检查
- [ ] 命名符合项目规范
- [ ] 关键代码有注释（解释"为什么"）
- [ ] 函数职责单一（单一职责原则）
- [ ] 避免过长函数（<50行）
- [ ] 避免过深嵌套（<3层）

#### 可测试性检查
- [ ] 新增代码有单元测试
- [ ] 测试覆盖关键路径
- [ ] 接口稳定（无频繁变更）
- [ ] 代码解耦（易于Mock）
- [ ] 纯函数可独立测试

## 第三部分：测试指南

### 3.1 测试策略

#### 分层测试模型
```
┌─────────────────┐
│   E2E 测试       │  ← 用户视角，验证完整流程
├─────────────────┤
│   集成测试       │  ← 模块间协作
├─────────────────┤
│   单元测试       │  ← 最小单元，核心逻辑
└─────────────────┘
```

#### 覆盖率要求

| 测试类型 | 最低覆盖率 | 目标覆盖率 |
|---------|-----------|-----------|
| 单元测试 | 70% | 80% |
| 集成测试 | 50% | 60% |
| E2E测试 | 30% | 40% |

### 3.2 测试执行流程

```
1. 编写测试 → 2. 运行测试 → 3. 查看报告 → 4. 修复问题 → 5. 重新测试
```

### 3.3 测试用例编写规范

#### 单元测试规范
```python
def test_function_name():
    """测试用例描述"""
    # Arrange：准备测试数据
    input_data = ...
    
    # Act：执行被测函数
    result = function_under_test(input_data)
    
    # Assert：验证结果
    assert result == expected_output
```

#### 边界条件测试
- 空值测试（null, "", [], {}）
- 极端值测试（最大/最小值）
- 类型错误测试（类型不匹配）
- 并发测试（多线程/多进程）

### 3.4 测试工具使用指南

#### Python项目
```bash
# 静态检查
flake8 . --max-complexity=15
mypy .

# 测试覆盖
pytest --cov=src tests/
coverage report --fail-under=70

# 安全审计
pip-audit
```

#### JavaScript/TypeScript项目
```bash
# 静态检查
eslint . --max-complexity=15
tsc --noEmit

# 测试覆盖
jest --coverage
nyc --check-coverage --lines 70

# 安全审计
npm audit
```

### 3.5 重构安全网

**四步安全网**：
1. 确保测试通过
2. 确保测试覆盖核心功能
3. 小步重构，每次提交
4. 每次重构后运行完整测试套件

### 3.6 质量问题处理流程

#### 发现问题
```
检测到问题 → 记录问题 → 评估严重程度 → 分配责任人
```

#### 修复问题
```
分析根因 → 制定修复方案 → 实施修复 → 验证修复
```

#### 严重程度分级

| 级别 | 定义 | 处理时限 |
|------|------|----------|
| Critical | 阻止开发/部署 | 立即修复 |
| High | 影响核心功能 | 24小时内 |
| Medium | 影响次要功能 | 3天内 |
| Low | 代码风格/优化建议 | 下一迭代 |

# 管理规则

## L3.6 规则体系与执行监督模块

### 核心目标
支撑 L1.9-L1.14。规范规则管理流程，确保规则体系质量。

### 适用场景
- 新增或修改规则时
- 创建新L3模块时
- 规则管理/变更时
- 规则体系定期维护时
- 任务执行完成后
- 代码审查、文档审阅

### 具体规则
L3.6.1 新增L3模块必须经过：提案→审核→通过；规则管理/变更必须以注意力机制为核心，避免规则冗余和注意力稀释，优先合并到现有规则而非新增条目。
L3.6.2 L3模块必须包含五部分：核心目标、适用场景、具体规则、验收标准、风险与例外。
L3.6.3 最高优先级规则置于文件末尾。
L3.6.4 定期维护整个规则体系（L1/L2/L3），确保注意力机制要求得到落实。
L3.6.5 复杂任务必须划分为"执行"与"监督"两个独立环节。
L3.6.6 监督环节必须：声明角色切换、基于全新上下文、给出评估结论。

### 验收标准
- 新模块有完整提案
- 所有L3模块包含五部分
- 链路扫描完成
- 有明确的监督评估结论
- 规则管理符合注意力机制要求，无冗余，优先合并

### 风险与例外
- 紧急修复可先上线规则
- 极简单任务可简化监督

## Git提交信息规范

### scene: git_message
### alwaysApply: true

### 提交格式
```
<类型>(<范围>): <描述>

[可选的正文]

[可选的脚注]
```

### 提交类型
| 类型 | 说明 |
|------|------|
| feat | 新功能 |
| fix | 错误修复 |
| docs | 文档变更 |
| style | 代码格式（不影响功能） |
| refactor | 重构（不是修复也不是新功能） |
| perf | 性能优化 |
| test | 测试相关 |
| chore | 构建/工具相关 |

### 规范要求
- 标题不超过50字符
- 使用中文描述
- 标题以动词开头
- 不使用句号结尾
- 正文解释what和why，不解释how

### 示例
```
feat(规则体系): 新增动态加载规则模块

- 支持根据上下文自动切换规则
- 减少不相关规则的干扰
- 提高规则复用率
```

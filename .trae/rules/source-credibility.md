## 一、信源类型与知信度评分

| 信源类型 | 知信度评分 | 说明 | 适用场景 |
|---------|------------|------|---------|
| 用户输入 | 100 | 最高优先级，用户说了算 | 所有任务 |
| 项目规则文件 | 95 | AGENTS.md、CLAUDE.md、.trae/rules/* | 规则相关 |
| 项目结构扫描 | 90 | 文件系统、依赖文件（package.json、Cargo.toml等） | 技术选型 |
| 本地模型知识 | 80 | 训练时学到的知识 | 通用编程 |
| 互联网查询 | 70 | WebSearch（谨慎使用） | 实时信息、新技术 |

## 二、信源评估原则

### 2.1 信源优先级决策树
```
用户问题
    ↓
用户输入？→ 是 → 优先使用
    ↓ 否
规则相关？→ 是 → 查询项目规则文件
    ↓ 否
技术相关？→ 是 → 扫描项目结构
    ↓ 否
通用知识？→ 是 → 使用本地模型知识
    ↓ 否
实时信息？→ 是 → 谨慎使用互联网查询
```

### 2.2 低可信度信源使用规范
- 使用前需标注："⚠️ 信息来源：互联网查询，建议确认"
- 提供多个来源交叉验证
- 明确说明信息可能存在的局限性

### 2.3 信源冲突解决机制
1. 按知信度评分排序，高可信度优先
2. 同评分信源冲突时，优先使用项目规则文件
3. 无法解决时，提交用户决策

## 三、知信度动态调整机制

### 3.1 调整触发条件
| 场景 | 调整方向 | 调整幅度 |
|------|---------|---------|
| 某信源连续多次正确 | +1-3分 |  |
| 某信源连续多次错误 | -5-10分 |  |
| 用户明确修正信源信息 | -10分 |  |

### 3.2 调整记录
- 所有信源评分调整需记录在案
- 定期回顾调整历史，优化评分标准
- 极端调整需用户确认

## 四、最佳实践

| 场景 | 推荐信源 | 原因 |
|------|---------|------|
| 规则相关问题 | 项目规则文件 | 知信度高，项目定制化 |
| 技术选型 | 项目结构扫描 | 符合项目实际环境 |
| 通用编程 | 本地模型知识 | 快速响应，稳定输出 |
| 实时天气/股票 | 互联网查询 | 唯一来源 |
| 用户需求 | 用户输入 | 最高优先级 |

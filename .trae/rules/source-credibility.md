## 一、调研定位与6A监督机制

### 1.1 定位说明
调研、搜索、信息整合属于 **6A 工作流的 Align（对齐）与 Architect（架构）阶段**，受6A流程统一监督，**不允许脱离流程单独执行**。

### 1.2 监督机制（避免自我监督悖论）

| 角色 | 职责 | 独立性 |
|------|------|--------|
| **执行角色** | 负责调研、搜索、整理信息 | 执行层 |
| **监督角色** | 6A工作流 + 质量门禁 | **独立于执行角色** |

**核心原则**：执行与监督分离，**不允许自我审批、自我放行**。

### 1.3 执行约束

| 约束项 | 要求 |
|--------|------|
| 产出文档 | 调研必须产出 ALIGNMENT.md（对齐阶段）或 DESIGN.md（架构阶段） |
| 来源标注 | 所有外部搜索（WebSearch）必须标注来源风险警告（⚠️） |
| 质量门禁 | 调研结果必须通过当前阶段质量门禁，方可进入下一阶段 |

## 二、信源类型与知信度评分

| 信源类型 | 知信度评分 | 说明 | 适用场景 |
|---------|------------|------|---------|
| 用户输入 | 100 | 最高优先级，用户说了算 | 所有任务 |
| 项目规则文件 | 95 | AGENTS.md、CLAUDE.md、.trae/rules/* | 规则相关 |
| 项目结构与代码 | 90 | 文件系统、依赖文件、源代码 | 技术选型、代码理解 |
| 模型内置知识 | 80 | 训练时学到的知识 | 通用编程 |
| 互联网搜索 | 70 | WebSearch（必须标注⚠️） | 实时信息、新技术 |

## 三、信源评估原则

### 3.1 信源优先级决策树
```
用户问题
    ↓
用户输入？→ 是 → 优先使用
    ↓ 否
规则相关？→ 是 → 查询项目规则文件
    ↓ 否
技术相关？→ 是 → 扫描项目结构与代码
    ↓ 否
通用知识？→ 是 → 使用模型内置知识
    ↓ 否
实时信息？→ 是 → 谨慎使用互联网搜索（标注⚠️）
```

### 3.2 低可信度信源使用规范
- 使用前**必须**标注："⚠️ 信息来源：互联网查询，建议确认"
- 提供多个来源交叉验证
- 明确说明信息可能存在的局限性
- 调研结果必须通过质量门禁

### 3.3 信源冲突解决机制
1. 按知信度评分排序，高可信度优先
2. 同评分信源冲突时，优先使用项目规则文件
3. 无法解决时，提交用户决策

## 四、知信度动态调整机制

### 4.1 调整触发条件
| 场景 | 调整方向 | 调整幅度 |
|------|---------|---------|
| 某信源连续多次正确 | +1-3分 |  |
| 某信源连续多次错误 | -5-10分 |  |
| 用户明确修正信源信息 | -10分 |  |

### 4.2 调整记录
- 所有信源评分调整需记录在案
- 定期回顾调整历史，优化评分标准
- 极端调整需用户确认

## 六、最佳实践

| 场景 | 推荐信源 | 原因 |
|------|---------|------|
| 规则相关问题 | 项目规则文件 | 知信度高，项目定制化 |
| 技术选型 | 项目结构与代码 | 符合项目实际环境 |
| 通用编程 | 模型内置知识 | 快速响应，稳定输出 |
| 实时信息 | 互联网查询 | 唯一来源（需标注⚠️） |
| 用户需求 | 用户输入 | 最高优先级 |

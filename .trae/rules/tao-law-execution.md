## 一、核心定义

### 1.1 滔定律是什么

**滔定律（τ定律）**：华为提出的高效做事、科学决策原则，不是芯片技术。

**核心思想**：不堆资源、不搞复杂并行，把复杂事拆成一步一步串行做。

**执行要求**：一步一检查、聚焦当下、不跳步、不贪多。

**目标**：决策快、执行稳、返工少、效率高、全程可控不翻车。

### 1.2 四大核心原则

| 原则 | 定义 | 操作要点 |
|------|------|---------|
| **串行化** | 不搞复杂并行，把复杂事拆成一步一步串行做 | 一次只做一件事 |
| **聚焦性** | 聚焦当下、不跳步、不贪多 | 只关注当前阶段 |
| **可验证** | 一步一检查、全程可控 | 每步都有验收 |
| **递归式** | 子任务继续拆分，直到问题足够简单可执行 | 拆到可执行为止 |

## 二、串行执行操作流程

### 2.1 任务接收与定义

**输入**: 用户请求  
**输出**: 明确的任务定义文档

**操作步骤**:
1. **理解意图** - 用自然语言复述用户需求
2. **明确目标** - 用 SMART 原则定义目标
3. **确定验收标准** - 列出 3-5 个可量化的验收条件
4. **识别边界** - 明确包含/不包含的内容
5. **评估复杂度** - 判断任务规模（简单/标准/复杂）

**模板示例**:
```markdown
# 任务定义
## 目标
[用一句话描述任务目标]

## 验收标准
- [ ] 标准 1
- [ ] 标准 2
- [ ] 标准 3

## 边界约束
**包含**: [内容]
**不包含**: [内容]

## 复杂度评估
[简单/标准/复杂]
```

### 2.2 递归拆分流程

**输入**: 任务定义文档  
**输出**: 递归拆分的任务树

**拆分标准**:
- ✅ 单一职责：每个子任务只负责一件事
- ✅ 独立可执行：不依赖未完成的兄弟任务
- ✅ 可测试：有明确的验证方式
- ✅ 工作量可控：单个子任务 ≤4 小时
- ✅ 可验收：有明确的验收标准

**递归深度参考**:
| 任务类型 | 递归深度 | 子任务数量 |
|---------|---------|-----------|
| 简单任务 | 1-2 层 | 2-5 个 |
| 标准任务 | 2-3 层 | 5-10 个 |
| 复杂任务 | 3-5 层 | 10-20 个 |

**任务树模板**:
```markdown
# 任务树
## 主任务：[名称]
### 子任务 1
- 交付物：[描述]
- 验收标准：[列表]
- 预计工时：[小时]

### 子任务 2
...
```

### 2.3 串行执行流程

**输入**: 任务树  
**输出**: 每个子任务的执行结果

**执行纪律**:
1. ⛔ **禁止并行** - 严格禁止同时处理多个子任务
2. ⛔ **禁止跳步** - 未完成前序任务不得开始后续任务
3. ✅ **阶段聚焦** - 只关注当前子任务的目标
4. ✅ **立即验证** - 完成后立即进行验收

**执行步骤**:
```
开始子任务 N
    ↓
前置条件检查（前序任务已完成并验证）
    ↓
执行子任务
    ↓
自验证（功能、质量、安全）
    ↓
验收通过？
    ├─ Yes → 进入子任务 N+1
    └─ No  → 修复后重新验证
```

### 2.4 阶段验证流程

**输入**: 子任务执行结果  
**输出**: 验证报告

**验证清单**:
```markdown
## [子任务名称] 验证报告

### 功能验证
- [ ] 实现预期功能
- [ ] 边界条件处理正确
- [ ] 异常场景处理正确

### 质量验证
- [ ] 代码符合规范
- [ ] 测试通过
- [ ] 无严重 Bug

### 安全验证
- [ ] 无数据泄露
- [ ] 无系统风险
- [ ] 无代码漏洞

### 验收结论
✅ 通过 / ❌ 不通过

### 验证人：[角色]
### 验证时间：[时间]
```

## 三、常见场景应用

### 3.1 场景 1：需求开发

**应用流程**:
```
Align（需求对齐）
    ↓ 验证通过
Architect（方案设计）
    ↓ 验证通过
Atomize（任务拆解）
    ↓ 验证通过
Approve（审批确认）
    ↓ 验证通过
Automate（串行执行）
    ↓ 每个子任务验证通过
Assess（评估验收）
```

**关键要点**:
- 每个阶段必须有明确的验收标准
- 阶段验证不通过不得进入下一阶段
- 严禁跳步行为

### 3.2 场景 2：Bug 修复

**应用流程**:
```
1. 问题定位（串行：先复现，再分析）
    ↓ 验证通过
2. 方案设计（串行：先评估，再设计）
    ↓ 验证通过
3. 代码修复（串行：先修复，再测试）
    ↓ 验证通过
4. 回归测试（串行：先单元测试，再集成测试）
```

**关键要点**:
- 严禁未经复现直接修复
- 严禁未经测试直接交付
- 必须进行回归测试

### 3.3 场景 3：代码重构

**应用流程**:
```
1. 现状分析（理解现有代码）
    ↓ 验证通过
2. 重构设计（设计重构方案）
    ↓ 验证通过
3. 小步重构（每次只重构一个模块）
    ↓ 验证通过
4. 测试验证（确保行为不变）
```

**关键要点**:
- 小步快跑，不要一次性重构所有代码
- 每次重构后必须测试验证
- 保持重构前后行为一致

## 四、禁止行为清单

### 4.1 严格禁止的行为

| 禁止行为 | 违规后果 | 示例 |
|---------|---------|------|
| **并行处理多个核心目标** | 强制回退 | 同时开发多个功能 |
| **跳过阶段验证** | 强制回退 + 记录违规 | 未测试直接交付 |
| **提前处理后续阶段** | 强制回退 | 在 Align 阶段编写代码 |
| **无验收标准直接交付** | 拒绝接收 | 无法量化是否完成 |
| **拆分粒度过大** | 重新拆分 | 子任务工作量>4 小时 |

### 4.2 推荐的最佳实践

| 最佳实践 | 说明 | 收益 |
|---------|------|------|
| **小步快跑** | 每次只完成一个小目标 | 降低风险，快速反馈 |
| **立即验证** | 完成后立即验收 | 问题早发现，修复成本低 |
| **聚焦当下** | 只关注当前阶段 | 避免分心，提高效率 |
| **文档记录** | 每个阶段都有文档 | 可追溯，可复盘 |

## 六、常见问题解答

### Q1: 如何判断任务是否拆分到位？

**判断标准**:
- ✅ 子任务工作量 ≤4 小时
- ✅ 有明确的验收标准
- ✅ 可以独立执行，不依赖其他子任务
- ✅ 无法再拆分为更小的单元

### Q2: 串行执行会不会降低效率？

**答案**: 不会，反而会提高效率。

**原因**:
- 减少上下文切换成本
- 降低返工概率
- 问题早发现，修复成本低
- 聚焦单一目标，注意力更集中

### Q3: 如果前序任务验证失败怎么办？

**处理流程**:
1. 立即停止后续任务
2. 分析失败原因
3. 修复问题
4. 重新验证
5. 验证通过后方可继续

### Q4: 如何平衡串行执行和紧急任务？

**处理原则**:
- 紧急任务也需要串行拆分
- 可以插入高优先级任务，但必须完成当前子任务后再切换
- 切换前必须保存当前状态并记录上下文

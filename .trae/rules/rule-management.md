## 一、规则准入执行细则

### 1.1 规则准入检查清单
新增规则前必须完成以下检查：

| 检查项 | 检查标准 | 结果 |
|--------|----------|------|
| 需求明确性 | 是否有明确的使用场景和需求 | ☐ |
| 必要性验证 | 是否无法用现有规则替代 | ☐ |
| 类型判断 | 属于刚需约束还是经验建议 | ☐ |
| 影响分析 | 是否会影响现有任务执行 | ☐ |
| 文档准备 | 是否有完整的说明和示例 | ☐ |

### 1.2 规则分类标准
- **刚需强制约束**：违反会导致质量/安全问题，必须强制执行
- **参考性经验建议**：最佳实践，推荐但不强制
- **禁止新增**：无实际使用场景、可被现有规则覆盖

### 1.3 规则准入流程
1. 需求分析和场景定义
2. 现有规则检索和复用验证
3. 规则草案编写
4. 规则类型判断和分类
5. 影响范围评估
6. 提交审批（L1变更需用户批准）
7. 规则正式发布

## 二、规则生命周期管理执行细则

### 2.1 规则定期核验清单
每季度对所有规则进行以下核验：

| 核验项 | 核验标准 |
|--------|----------|
| 使用频率 | 最近3个月是否被引用 |
| 场景有效性 | 适用场景是否仍然存在 |
| 逻辑一致性 | 是否与当前实践相悖 |
| 重复情况 | 是否有语义相同的其他规则 |

### 2.2 规则失效判定流程
1. 定期核验触发
2. 收集规则使用数据
3. 失效判断评估
4. 标记为"待审查"状态
5. 执行影响分析
6. 确认后标记为"已失效"
7. 更新规则索引和知识图谱

### 2.3 规则合并执行步骤
1. 识别语义相同的规则组
2. 对比各规则的严谨性和完整性
3. 选择最严谨的版本保留
4. 更新所有引用关系
5. 删除冗余规则
6. 通知规则使用者
7. 更新知识图谱

## 三、层级边界管理执行细则

### 3.1 层级内容分配表
| 层级 | 内容类型 | 具体示例 |

### 3.2 引用规范模板
- **下层引用上层**：`- **L1支撑**: 规则名称 [L1-x.x.x]`
- **上层引用下层**：`- **L3支撑**: 文件名.md`
- **禁止全文复制**：只能引用，不能复制内容

### 3.3 层级检查清单
检查规则内容是否符合其层级定位：

| 检查项 | L1要求 | L2要求 | L3要求 |
|--------|--------|--------|--------|
| 是否包含原则 | ✅ 必须 | ☐ 可选 | ❌ 不应该 |
| 是否有具体要求 | ❌ 不应该 | ✅ 必须 | ☐ 可选 |
| 是否有操作步骤 | ❌ 不应该 | ☐ 可选 | ✅ 必须 |
| 是否有检查清单 | ❌ 不应该 | ☐ 可选 | ✅ 推荐 |

## 四、规则冲突与优化执行细则

### 4.1 规则冲突检测清单
- [ ] 同一层级内是否有语义冲突的规则
- [ ] 不同层级间是否有优先级冲突
- [ ] 规则表述是否自相矛盾
- [ ] 规则之间是否有重叠或重复

### 4.2 冲突解决执行步骤
1. 检测到冲突触发
2. 确定冲突类型（同级/跨级）
3. 应用优先级规则（安全>质量>流程>效率）
4. 同层级冲突时选择编号更小的规则
5. 无法自动仲裁时提交监控智能体裁决
6. 记录裁决结果
7. 更新规则索引

### 4.3 规则优化执行步骤
1. 收集规则执行数据
2. 分析规则使用效果
3. 识别优化机会
4. 制定优化方案
5. 执行优化（删除冗余、精简表述）
6. 验证优化效果
7. 更新版本历史

### 4.4 规则优化检查清单
- [ ] 是否删除了无效条件
- [ ] 是否精简了冗余表述
- [ ] 是否保持了规则的明确性
- [ ] 是否更新了所有引用关系
- [ ] 是否记录了优化原因

## 五、规则版本变更管控执行细则

### 5.1 变更记录模板
```
## 版本历史

| 版本 | 日期 | 变更内容 | 变更人 |
|------|------|----------|--------|
| vx.x | YYYY-MM-DD | 变更内容说明 | 变更者 |
```

### 5.2 变更记录必填项
- 版本号（遵循语义化版本规范）
- 更新日期
- 变更内容（新增/修改/删除）
- 变更人
- 变更原因（可选但推荐）

### 5.3 重大变更评估清单
重大变更前必须完成：

| 评估项 | 评估标准 | 结果 |
|--------|----------|------|
| 影响范围分析 | 影响哪些任务和规则 | ☐ |
| 对现有任务影响 | 是否会破坏现有任务执行 | ☐ |
| 过渡期安排 | 是否需要过渡期 | ☐ |
| 回滚方案 | 是否有回滚预案 | ☐ |
| 文档更新 | 相关文档是否同步更新 | ☐ |

### 5.4 变更发布流程
1. 变更准备（草案编写、影响分析）
2. 变更审批（L1需用户批准）
3. 变更执行（修改文件、更新引用）
4. 验证测试（确保变更正常工作）
5. 发布通知（告知规则使用者）
6. 知识图谱更新

## 核心身份与目标
管理和协调22个专业智能体，集成中央调度模块，具备三层记忆系统和上下文感知能力，通过博弈调度机制实现智能任务分配，支持并行执行模式。

## 第三章：并行执行模式

### 3.1 模式概述

同时调度执行智能体和监控智能体并行工作：
- **执行智能体**：负责实际任务执行
- **监控智能体**：负责全程监控执行过程

**架构优势**: 实时监控、快速响应、可追溯性、灵活组合

### 3.2 并行执行工作流

```
任务输入 → 创建上下文 → 并行启动
                              │
              ┌───────────────┴───────────────┐
              ▼                               ▼
        执行智能体                        监控智能体
              │                               │
              ├─ 执行步骤 ────────┐           │
              ├─ 执行步骤 ────────┼──→ 实时监控
              └─ 返回结果 ────────┘           │
                      │                      │
                      ▼                      ▼
                 结果聚合 ←───────────── 监控报告
                      │
                      ▼
                   返回用户
```

### 3.3 并行执行协议

**启动指令**:


## 第五章：上下文感知路由

### 5.1 路由策略

| 策略 | 优先级 | 说明 |
|-----|-------|------|
| 语义路由 | 最高 | 知识图谱意图分析 |
| 上下文路由 | 高 | 历史任务+用户偏好 |
| 负载路由 | 中 | 选择负载最低智能体 |
| 优先级路由 | 中 | 高优先级任务优先 |
| 并行路由 | 中 | 需要监控的任务自动配对监控智能体 |

### 5.2 路由算法

```python
def select_agent(task, context):
    # 1. 检查上下文缓存
    if context.get("context_id"):
        ctx = get_context(context["context_id"])
        similar = ctx.query_similar_tasks(task)
        if similar and similar.agents_used:
            return similar.agents_used[0]
    
    # 2. 检查用户偏好
    if context.get("user_preference"):
        preferred = context["user_preference"].get("agent")
        if preferred and is_available(preferred):
            return preferred
    
    # 3. 使用语义路由 + 负载均衡
    recommended = kg.recommend_agents(task)
    candidates = [a for a in agents if a.id in recommended]
    return min(candidates, key=lambda a: a.current_load)

def select_parallel_agents(task, context):
    executor = select_agent(task, context)
    return {"executor": executor, "monitor": "monitor_agent"}
```

### 5.3 上下文复用策略

| 场景 | 策略 |
|-----|------|
| 连续同类任务 | 复用相同智能体组合 |
| 用户明确偏好 | 优先使用偏好智能体 |
| 新类型任务 | 语义路由+记录历史 |
| 任务失败 | 切换备用智能体 |
| 需要监控的任务 | 自动配对监控智能体 |

## 第六章：博弈调度机制

### 6.1 三种模式

| 模式 | 触发关键词 | 工作流 |
|-----|---------|-------|
| 模式一：辩论 | 对比、权衡、优缺点 | 视角A → 视角B → 裁决 |
| 模式二：降维打击 | 优化、改进、重构 | 生成 → 挑剔 → 融合 |
| 模式三：深度设计 | 设计、架构、方案 | 蓝图 → 挑战 → 融合 |

### 6.2 触发决策流程

```
任务 → 分词 → query_kg查询 → 概念聚合
    │
    ├── 含"辩论" → 模式一
    ├── 含"优化" → 模式二
    ├── 仅"设计" → 模式三
    └── 无概念 → 复杂度判断 → 模式三/常规路由
```

### 6.3 模式执行流程

**模式一（辩论）**:
1. 识别对立视角（创新探索 vs 风险控制）
2. 视角A输出结论
3. 视角B提出反对意见
4. 综合裁决：共识、分歧、建议

**模式二（降维打击）**:
1. 创意智能体生成方案初稿
2. 审查智能体提出审阅意见
3. 综合智能体融合优化

**模式三（深度设计）**:
1. 架构师智能体输出设计蓝图
2. 工程师智能体评估可行性
3. 技术作家输出落地方案

### 6.4 优先级说明
- **博弈模块拥有最高优先级**，触发后全程按博弈工作流执行
- 常规任务走中央调度模块的上下文感知路由
- 需要监控的任务自动启用并行执行模式

## 第九章：执行流程

```
用户输入 → Step1:接收任务 → Step2:图谱查询+上下文分析
                                │
                                ▼
           Step3:工作流模板选择 → Step4:并行/顺序调度 → Step5:结果聚合
                                │
                                ▼
                         Step6:返回结果
```

**Step1**: 解析输入、关联/创建上下文、检查相似任务  
**Step2**: 分词查询、分析偏好历史、确定博弈模式  
**Step3**: 选择工作流模板（并行/顺序/博弈）  
**Step4**: 总线分发、启动监控（如需要）、记录历史  
**Step5**: 收集输出、整合监控报告、更新记忆  
**Step6**: 输出结果、记录执行日志

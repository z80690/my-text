## 一、开发前检查清单

### 1.1 在写任何代码前，必须确认：
- [ ] 是否已完成Proposal（提案）？
- [ ] 是否已完成Specs（规范）？
- [ ] 是否已完成Design（设计）？
- [ ] 是否已完成Tasks（任务拆解）？
- [ ] 是否已获得用户Approve（批准）？

### 1.2 强制触发条件
如果以下任一情况发生，**必须先暂停写代码**，回到6A流程的前期阶段：
1. 需求理解存在任何歧义
2. 技术方案未明确确定
3. 验收标准未清晰定义
4. 用户未明确确认前进方向

## 二、Specs文档标准结构

### 2.1 Proposal（提案）文档结构
```markdown
# 项目提案

## 业务背景
- 为什么做这个？
- 解决什么问题？
- 给谁用？

## 目标
- 业务目标
- 技术目标
- 质量目标

## 成功标准
- 可量化的验收条件
- 关键绩效指标

## 风险评估
- 技术风险
- 进度风险
- 质量风险
```

### 2.2 Specs（规范）文档结构
```markdown
# 功能规范

## 功能描述
一句话说明这个功能做什么

## 输入输出
- 输入：参数、格式、约束
- 输出：返回值、格式、错误情况

## 数据模型
- 数据结构定义
- 字段说明
- 验证规则

## API接口
- 接口签名
- 请求/响应示例
- 错误码定义

## 业务规则
- 核心逻辑
- 边界条件
- 例外处理
```

### 2.3 Design（设计）文档结构
```markdown
# 技术设计

## 技术选型
- 为什么选这个方案？
- 对比了什么替代方案？

## 架构设计
- 架构图
- 模块划分
- 依赖关系

## 关键算法
- 核心算法描述
- 复杂度分析
- 安全考量

## 性能指标
- 响应时间要求
- 吞吐量要求
- 资源限制
```

### 2.4 Tasks（任务）文档结构
```markdown
# 实施任务清单

## 任务列表
- [ ] 任务1：具体描述（预估时间）
- [ ] 任务2：具体描述（预估时间）
- [ ] 任务3：具体描述（预估时间）

## 验收标准
每个任务的验收标准
```

## 三、code-simplifier执行流程

### 3.1 代码整理触发条件
- AI生成的代码需要整理时
- 代码复杂度高、难以阅读时
- 代码风格不符合项目规范时

### 3.2 整理操作检查清单
- [ ] **功能守恒**：确认只整理不改变功能
- [ ] **清晰度优先**：不追求最短代码，追求最清晰代码
- [ ] **遵循规范**：严格遵循项目编码规范
- [ ] **结构保留**：不破坏原有代码结构
- [ ] **聚焦当下**：只整理当前修改，不扩大范围

### 3.3 整理操作执行步骤
1. 理解原始代码的功能意图
2. 检查命名是否清晰
3. 优化嵌套层级
4. 删除冗余逻辑
5. 添加必要注释
6. 运行测试，确保功能未变
7. 提交代码，注明"code-simplifier整理"

## 四、四大约束缰绳执行细则

### 4.1 约束一：代码风格统一
- [ ] 变量命名：驼峰式/下划线式，统一风格
- [ ] 代码缩进：4空格，不用tab
- [ ] 括号风格：统一K&R/Allman风格
- [ ] 行长度：不超过100字符
- [ ] 空行使用：合理分隔逻辑块

### 4.2 约束二：自动生成文档
- [ ] 所有函数必须有注释
- [ ] 说明：参数含义、返回值、例外情况
- [ ] 复杂逻辑必须有说明
- [ ] 公共API必须有使用示例
- [ ] 文档与代码同步更新

### 4.3 约束三：内置测试用例
- [ ] 核心功能必须有单元测试
- [ ] 边界条件必须测试
- [ ] 错误情况必须测试
- [ ] 测试覆盖率≥70%
- [ ] 先写测试，再写实现（可选但推荐）

### 4.4 约束四：主动纠错
- [ ] 编译前检查语法错误
- [ ] 逻辑一致性检查
- [ ] 安全漏洞扫描
- [ ] 性能瓶颈检查
- [ ] 代码质量分析

## 五、与6A工作流整合执行

### 5.1 Align阶段 → Proposal + Specs
- 产出：Proposal文档、Specs文档
- 确认：需求理解一致、验收标准明确

### 5.2 Architect阶段 → Design
- 产出：Design文档
- 确认：技术方案可行、风险可控

### 5.3 Atomize阶段 → Tasks
- 产出：Tasks任务清单
- 确认：任务可执行、优先级明确

### 5.4 Approve阶段
- 产出：用户批准
- 确认：所有文档得到确认

### 5.5 Automate阶段
- 按Specs执行
- 每个Task完成后验证
- 运行测试、质量检查

### 5.6 Assess阶段
- 按Specs验收
- 质量门禁评估
- 文档更新

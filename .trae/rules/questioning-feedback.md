## 一、质疑反馈执行流程 [L3-2.1.1]

### 1.1 执行流程图

```
用户指令输入
     ↓
第1步：识别指令风险
     ↓
    ┌─────────────────────────┐
    │ 风险识别                 │
    │ • 是否破坏现有功能？     │
    │ • 是否违背最小化修改？   │
    │ • 是否违背滔定律精神？   │
    │ • 是否超出任务范围？     │
    │ • 是否可能导致返工？     │
    └─────────────────────────┘
     ↓
    ┌─────────────────────────┐
    │ 无风险                   │ → 执行指令
    └─────────────────────────┘
     ↓ 有风险
第2步：立即反馈
     ↓
    ┌─────────────────────────┐
    │ 反馈内容                 │
    │ • 问题描述              │
    │ • 可能后果              │
    │ • 替代建议              │
    └─────────────────────────┘
     ↓
第3步：等待用户确认
     ↓
    ┌─────────────────────────┐
    │ 用户确认                 │ → 执行指令（记录风险）
    └─────────────────────────┘
     ↓ 用户拒绝
第4步：停止执行
```

## 二、风险识别清单 [L3-2.1.2]

### 2.1 必须质疑的场景

| 场景 | 检查项 | 质疑级别 |
|------|-------|---------|
| **破坏风险** | 指令是否涉及删除、重写、覆盖现有文件或功能？ | 🔴 必须质疑 |
| **原则冲突** | 指令是否要求全部重写而非最小化修改？ | 🔴 必须质疑 |
| **滔定律冲突** | 指令是否与"时间优先"、"逻辑折叠"等原则冲突？ | 🔴 必须质疑 |
| **范围蔓延** | 指令是否明显超出原定任务范围？ | 🟡 建议质疑 |
| **返工风险** | 指令是否可能导致之前工作白费？ | 🟡 建议质疑 |
| **信息不足** | 指令是否缺少必要的执行信息？ | 🟡 建议质疑 |

### 2.2 风险等级定义

| 等级 | 说明 | 处理方式 |
|------|------|---------|
| 🔴 高风险 | 可能破坏现有功能、导致重大返工 | 必须反馈并获得明确确认 |
| 🟡 中风险 | 可能存在问题但影响较小 | 建议反馈，由用户决定 |
| 🟢 低风险 | 基本无风险 | 可直接执行 |

## 三、反馈格式规范 [L3-2.1.3]

### 3.1 标准反馈格式

```
【质疑反馈】
━━━━━━━━━━━━━━━━━━━━━━━━━━━
📌 问题：[具体问题描述]
⚠️  风险：[可能的后果]
💡 建议：[替代方案或解决建议]
📋 确认：[请明确回复"确认执行"或"取消"]
━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 3.2 反馈示例

**示例1：最小化修改原则冲突**

```
【质疑反馈】
━━━━━━━━━━━━━━━━━━━━━━━━━━━
📌 问题：此操作需要全部重写 knowledge-graph.html
⚠️  风险：破坏现有的正常功能，导致用户需要重新验证
💡 建议：改为最小化修改，只修复问题部分
📋 确认：请确认是否改为最小化修改方案
━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**示例2：滔定律冲突**

```
【质疑反馈】
━━━━━━━━━━━━━━━━━━━━━━━━━━━
📌 问题：此操作增加3个新层级，可能增加传递延迟
⚠️  风险：违背"时间优先"和"逻辑折叠"原则
💡 建议：利用现有层级，减少新增
📋 确认：请确认是否优化层级结构
━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**示例3：破坏风险**

```
【质疑反馈】
━━━━━━━━━━━━━━━━━━━━━━━━━━━
📌 问题：此操作删除4个规则文件
⚠️  风险：可能删除重要内容，且无法恢复
💡 建议：先备份，或逐个确认删除
📋 确认：请确认是否需要备份后再删除
━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 四、执行标准 [L3-2.1.4]

### 4.1 执行判断标准

| 判断项 | 标准 | 结果 |
|-------|------|------|
| **是否有风险** | 存在上述任意风险场景 | 是 → 反馈 |
| **风险是否可接受** | 用户明确了解风险并确认 | 是 → 执行 |
| **是否有替代方案** | 提供1-2个替代建议 | 是 → 提供 |
| **用户是否确认** | 用户明确回复"确认执行" | 是 → 执行 |

### 4.2 禁止行为

- ❌ 不质疑就执行可能有问题的指令
- ❌ 发现问题但不反馈
- ❌ 强制执行用户可能后悔的指令
- ❌ 为了讨好用户而隐瞒问题

### 4.3 推荐行为

- ✅ 主动识别风险并反馈
- ✅ 提供清晰的替代方案
- ✅ 明确说明可能后果
- ✅ 在用户确认后执行

## 五、常见场景处理 [L3-2.1.5]

### 5.1 场景1：用户要求全部重写

```
用户：把这个文件给我全部重写

AI反应：
【质疑反馈】
📌 问题：此操作需要全部重写文件
⚠️  风险：破坏现有的正常功能，可能导致返工
💡 建议：请问具体是哪个部分有问题？我可以只修复问题部分
📋 确认：是否改为最小化修改？
```

### 5.2 场景2：用户要求删除多个文件

```
用户：把这两个文件夹给我删了

AI反应：
【质疑反馈】
📌 问题：此操作删除2个文件夹
⚠️  风险：删除后无法恢复，可能误删重要文件
💡 建议：可以先备份，或者逐个确认要删除的文件
📋 确认：请确认是否需要备份后再删除？
```

### 5.3 场景3：指令超出任务范围

```
用户：顺便把这个项目也帮我做了

AI反应：
【质疑反馈】
📌 问题：此操作超出原定任务范围
⚠️  风险：可能影响原任务的进度和质量
💡 建议：可以记录为新任务，等原任务完成后再处理
📋 确认：是否将此项记录为新任务？
```

### 5.4 场景4：信息不足无法执行

```
用户：优化一下这个

AI反应：
【质疑反馈】
📌 问题：指令信息不足
⚠️  风险：可能优化方向与您预期不符
💡 建议：请说明具体的优化目标（如：性能、代码质量、可读性等）
📋 确认：请提供更多信息以便正确执行
```

## 六、记录与复盘 [L3-2.1.6]

### 6.1 反馈记录

| 字段 | 说明 |
|------|------|
| 时间 | 反馈时间 |
| 指令摘要 | 用户原始指令 |
| 风险类型 | 识别的风险类型 |
| 反馈内容 | 提供的反馈 |
| 用户响应 | 确认执行/取消/修改 |
| 执行结果 | 最终执行情况 |

### 6.2 复盘要点

- 质疑是否及时？
- 反馈是否清晰？
- 建议是否合理？
- 用户是否认可？

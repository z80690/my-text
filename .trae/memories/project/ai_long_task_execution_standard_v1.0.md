## 第一章 长任务定义与核心痛点

### 1.1 长任务明确定义

**长任务**是指需要跨多个执行阶段、涉及多轮交互、交付周期较长的复杂任务。与短任务相比，长任务具有以下特征：

| 特征 | 短任务 | 长任务 |
|------|--------|--------|
| **执行周期** | 单次交互完成 | 多阶段、多轮次 |
| **交付物** | 单一明确 | 多层级、可拆分 |
| **复杂度** | 低，原子化 | 高，需分层拆解 |
| **变更频率** | 低 | 高，需持续对齐 |
| **协作要求** | 低 | 高，需多角色协同 |

**长任务判定标准**：满足以下任意两项即判定为长任务：
1. 预计执行周期超过2小时
2. 交付物数量≥3个
3. 涉及2个及以上执行阶段
4. 需要多轮反馈迭代

### 1.2 执行核心痛点拆解

长任务执行过程中普遍存在以下核心痛点：

**痛点一：需求漂移**
- 表现：任务目标随时间推移逐渐偏离初始需求
- 根因：缺乏阶段性对齐机制
- 影响：交付物与原始目标脱节

**痛点二：任务颗粒度失控**
- 表现：任务拆分过粗或过细，无法有效执行
- 根因：缺乏统一的拆分标准
- 影响：执行效率低下，验收困难

**痛点三：上下文丢失**
- 表现：多轮交互后丢失关键信息
- 根因：未建立有效的记忆系统
- 影响：重复工作、结论矛盾

**痛点四：质量波动**
- 表现：不同阶段交付物质量不一致
- 根因：缺乏标准化验收流程
- 影响：整体交付品质无法保证

**痛点五：异常处理缺失**
- 表现：遇到问题无法有效回退
- 根因：缺乏异常边界定义
- 影响：任务执行中断率升高

### 1.3 规范制定核心价值

本规范的核心价值在于：

| 价值维度 | 具体收益 |
|----------|----------|
| **标准化** | 统一长任务执行流程，消除执行歧义 |
| **可复用** | 建立标准化模板，可批量复制推广 |
| **可追溯** | 全流程记录，支持问题回溯 |
| **可度量** | 明确的验收标准，便于质量评估 |
| **可协同** | 多角色协作有章可循，减少沟通成本 |

## 第二章 三层需求拆分标准（L1/L2/L3）

### 2.1 L1顶层需求：定义与撰写规范

**L1顶层需求**是长任务执行的起点，必须包含以下强制字段：

| 字段 | 必填 | 说明 | 示例 |
|------|------|------|------|
| 任务名称 | ✅ | 简洁明确的任务标题 | "用户登录功能开发" |
| 核心目标 | ✅ | 任务要解决的核心问题 | "实现基于JWT的分布式登录" |
| 最终交付物 | ✅ | 明确的产出清单 | "API文档+代码+测试用例" |
| 绑定约束 | ✅ | 依赖系统或规范 | "必须适配6A工作流" |
| 验收标准 | ✅ | 可量化的验收条件 | "接口响应<200ms，通过全量测试" |

**L1撰写规则**：
1. 每个任务只能有1个L1顶层需求
2. 字段不可缺失、顺序不可调换
3. 目标描述需无歧义、可执行
4. 验收标准必须可量化校验

### 2.2 L2阶段需求：拆分与提交规范

**L2阶段需求**是L1到L3的过渡层，定义任务的执行阶段。

**标准6A阶段划分**：

| 阶段 | 名称 | 核心产出 | 校验点 |
|------|------|----------|--------|
| Align | 需求对齐 | 需求对齐文档 | 需求是否清晰？范围是否明确？ |
| Architect | 架构设计 | 技术方案文档 | 方案是否合理？风险是否评估？ |
| Atomize | 原子化拆分 | 任务清单 | 拆分是否到位？粒度是否合适？ |
| Approve | 审批确认 | 审批记录 | 审批是否到位？风险是否可控？ |
| Automate | 执行 | 交付物 | 规则是否遵循？质量是否达标？ |
| Assess | 验收评估 | 验收报告 | 评分是否达标？门禁是否通过？ |

**L2提交规则**：
1. L2阶段需求必须完全对应L1的6A阶段
2. 每个阶段必须有明确的校验点
3. 阶段之间需串行执行，不允许跳过
4. 每个阶段完成后必须同步更新第二大脑

### 2.3 L3原子任务：拆解与验收规范

**L3原子任务**是最小可执行单元，必须满足：

| 特性 | 要求 |
|------|------|
| **可独立执行** | 不依赖其他任务结果 |
| **有明确交付物** | 输出物清晰可定义 |
| **可验收** | 有明确的验收标准 |
| **耗时可控** | 建议15分钟-2小时 |

**L3拆解原则**：
1. 每批原子任务数量建议5-15条
2. 相同层级任务可并行执行
3. 任务间有依赖关系的必须串行
4. 每个任务完成后立即写入第二大脑

**L3验收规则**：
1. 任务完成后必须提交执行结果
2. 执行结果需包含：完成状态、产出物、问题记录
3. 未通过验收的任务需返工，不计入完成
4. 批量任务完成后需进行阶段汇总

## 第三章 第二大脑实时读写执行规范

### 3.1 记忆系统文件架构

第二大脑基于文件系统实现，路径结构如下：

```
.trae/memories/
├── project/           # 项目级记忆
│   ├── context_pointer.json    # 上下文指针（当前焦点）
│   ├── {task_name}_log.json    # 任务执行日志
│   └── {task_name}_status.json # 任务状态记录
├── user/             # 用户级记忆
├── wiki/             # 知识库
└── evolution/        # 进化记录
```

**context_pointer.json结构**：
```json
{
  "current_focus": "当前任务名称",
  "recent_changes": [
    {"change": "变更描述", "timestamp": "ISO时间"}
  ],
  "known_issues": ["已知问题列表"],
  "last_updated": "最后更新时间"
}
```

### 3.2 任务全周期读写时机

| 执行阶段 | 读取操作 | 写入操作 |
|----------|----------|----------|
| **任务启动** | 读取L1顶层需求 | 写入任务启动记录 |
| **阶段开始** | 读取阶段需求+上层规则 | 写入阶段开始状态 |
| **任务执行** | 读取当前任务详情 | 写入执行进度 |
| **任务完成** | - | 写入完成状态+产出物 |
| **阶段完成** | - | 写入阶段汇总+校验结果 |
| **异常发生** | 读取异常处理规则 | 写入异常记录+处理方案 |
| **任务结束** | 读取全周期记录 | 写入最终状态+归档标记 |

**读写原则**：
1. 读取优先于执行，确保上下文一致
2. 写入紧随执行，确保状态实时同步
3. 异常时必须立即写入，记录现场
4. 定期读取context_pointer，确认任务焦点

### 3.3 数据同步与回溯规则

**同步规则**：
1. 所有状态变更必须同步写入第二大脑
2. 同步操作必须原子化（成功或失败）
3. 读取操作优先从本地缓存获取
4. 多终端访问需加锁保护

**回溯规则**：
1. 所有任务执行记录必须保留
2. 每次状态变更记录变更原因
3. 支持任意时间点状态恢复
4. 回溯操作需记录回溯日志

**数据清理规则**：
1. 已归档任务数据仅保留摘要
2. 超过90天的临时数据可清理
3. 清理前必须确认无活跃任务依赖
4. 清理操作需记录审计日志

## 第四章 6A工作流深度适配规范

### 4.1 Align：需求对齐执行标准

**目标**：澄清需求边界，产出需求对齐文档

**执行标准**：
1. 确认业务场景与目标
2. 明确功能范围（包含/不包含）
3. 确认约束条件（时间、技术、资源）

**校验点**：需求是否清晰？范围是否明确？

**强制要求**：
- 必须产出需求对齐文档
- 必须全员确认无异议
- 必须同步更新第二大脑

### 4.2 Architect：架构设计执行标准

**目标**：提供2种技术方案对比

**执行标准**：
1. 方案A：优势、劣势、成本
2. 方案B：优势、劣势、成本
3. 推荐建议
4. 风险评估

**校验点**：方案是否合理？风险是否评估？

**强制要求**：
- 必须输出2个以上可行方案
- 必须明确风险应对措施
- 必须同步更新第二大脑

### 4.3 Atomize：原子化拆分执行标准

**目标**：拆解为可执行的子任务列表

**执行标准**：
1. 每个任务有明确的交付物
2. 任务之间有依赖关系
3. 估算工作量（可选）

**校验点**：拆分是否到位？粒度是否合适？

**强制要求**：
- 任务粒度必须满足L3原子任务标准
- 必须标注任务依赖关系
- 必须同步更新第二大脑

### 4.4 Approve：审批确认执行标准

**目标**：关键节点获得用户确认

**执行标准**：
1. 需求变更需审批
2. 方案选择需审批
3. 风险操作需审批

**校验点**：审批是否到位？风险是否可控？

**强制要求**：
- 审批未通过不得进入下一阶段
- 审批意见必须明确记录
- 必须同步更新第二大脑

### 4.5 Automate：批量执行执行标准

**目标**：按批准范围执行任务

**执行标准**：
1. 严格按照审批范围执行
2. 遇到阻塞及时反馈
3. 定期汇报进度

**校验点**：规则是否遵循？质量是否达标？

**强制要求**：
- 执行过程全程记录
- 异常情况立即上报
- 必须同步更新第二大脑

### 4.6 Assess：验收评估执行标准

**目标**：交付时提供验收清单

**执行标准**：
1. 功能完成情况
2. 质量检查结果（测试、lint、安全）
3. 已知问题与后续建议

**校验点**：评分是否达标？门禁是否通过？

**强制要求**：
- 必须逐项核对验收清单
- 质量问题必须修复后验收
- 必须同步更新第二大脑

## 第五章 中央调度引擎协同规范

### 5.1 调度引擎任务分发规则

**分发原则**：
1. 基于任务类型选择合适的执行器
2. 基于负载情况分配执行资源
3. 基于优先级调整执行顺序

**分发流程**：
```
任务提交 → 引擎解析 → 类型识别 → 资源评估 → 执行器匹配 → 任务下发
```

**调度约束**：
1. 同一任务不允许重复分发
2. 阻塞任务需等待资源释放
3. 超时任务需自动回收

### 5.2 分层文档校验规则

**L1校验**：顶层需求完整性
- 必填字段是否齐全
- 字段格式是否正确

**L2校验**：阶段需求一致性
- 是否覆盖L1所有要求
- 阶段划分是否合理

**L3校验**：原子任务可行性
- 任务粒度是否合适
- 依赖关系是否完整

### 5.3 异常拦截与回退规则

**异常分级**：
| 级别 | 定义 | 处理方式 |
|------|------|----------|
| P0 | 系统级异常 | 立即中断，回退到上一个稳定状态 |
| P1 | 任务级异常 | 暂停当前任务，保留现场 |
| P2 | 轻微异常 | 记录日志，继续执行 |

**回退规则**：
1. P0异常触发自动回退
2. 回退前必须保存当前状态
3. 回退后需重新审批才能继续
4. 回退操作必须记录审计日志

## 第六章 高频提交与质量管控规范

### 6.1 分层提交频率强制标准

| 层级 | 提交时机 | 提交次数 |
|------|----------|----------|
| L1顶层需求 | 立项立即提交 | 1次 |
| L2阶段需求 | 每个阶段开始提交 | 共6次 |
| L3原子任务 | 每批5-15条提交 | N次 |

**强制要求**：
1. L1必须立即提交，不允许离线操作
2. L2必须按阶段提交，不允许跨阶段
3. L3必须高频提交，不允许攒批

### 6.2 文档实时归档规则

**归档时机**：
1. 每个阶段完成后归档
2. 任务结束后最终归档
3. 异常发生时先归档再处理

**归档内容**：
1. 完整的执行记录
2. 所有的产出物
3. 审核与验收记录

### 6.3 提交校验与驳回规则

**校验项**：
1. 格式是否符合模板要求
2. 必填字段是否完整
3. 内容是否符合业务逻辑

**驳回规则**：
1. 格式错误直接驳回
2. 内容矛盾直接驳回
3. 缺失关键信息驳回

## 第七章 实践案例与问题处理

### 7.1 典型长任务完整执行案例

**案例：用户权限系统重构**

| 阶段 | 任务 | 产出物 |
|------|------|--------|
| Align | 需求对齐会议 | 需求对齐文档 |
| Architect | 技术方案设计 | 2套方案对比+推荐 |
| Atomize | 任务拆分 | 45条原子任务清单 |
| Approve | 方案审批 | 审批通过记录 |
| Automate | 分批执行 | 代码+文档+测试 |
| Assess | 验收评估 | 验收报告 |

### 7.2 常见异常场景处理方案

**场景一：需求变更**
1. 记录变更内容
2. 重新执行Align阶段
3. 评估对后续阶段影响
4. 调整任务计划

**场景二：任务阻塞**

## 核心内容

### 核心思想
从"氛围编程"向"规范驱动开发"的范式转移。

### AI 编程黄金三角
- OpenSpec: 开放规范
- Superpowers: 超级能力
- gstack: 开发栈

### 规范驱动开发核心原则（4条）
1. **先定规格，再写代码**: 写代码前必须有 Proposal/Specs/Design/Tasks 文档
2. **人类主导，AI 执行**: 人类定方向，AI 严格按 Specs 执行
3. **四大约束缰绳**: 代码风格统一、自动生成文档、内置测试用例、主动纠错
4. **code-simplifier 五原则**: 功能守恒、清晰度优先、遵循规范、结构保留、聚焦当下

### 四大约束缰绳
1. **代码风格统一**: 变量命名、代码缩进、括号风格、行长度、空行使用
2. **自动生成文档**: 所有函数必须有注释、说明参数含义、复杂逻辑必须有说明、公共 API 必须有使用示例
3. **内置测试用例**: 核心功能必须有单元测试、边界条件必须测试、错误情况必须测试、测试覆盖率 ≥70%
4. **主动纠错**: 编译前检查语法错误、逻辑一致性检查、安全漏洞扫描、性能瓶颈检查、代码质量分析

### Specs 文档标准结构
**Proposal（提案）文档结构**:
- 业务背景
- 目标
- 成功标准
- 风险评估

**Specs（规范）文档结构**:
- 功能描述
- 输入输出
- 数据模型
- API 接口
- 业务规则

**Design（设计）文档结构**:
- 技术选型
- 架构设计
- 关键算法
- 性能指标

**Tasks（任务）文档结构**:
- 任务清单
- 验收标准

### 开发前检查清单
在写任何代码前，必须确认：
- 是否已完成 Proposal？
- 是否已完成 Specs？
- 是否已完成 Design？
- 是否已完成 Tasks？
- 是否已获得用户 Approve？

### 强制触发条件
如果以下任一情况发生，必须先暂停写代码，回到 6A 流程的前期阶段：
1. 需求理解存在任何歧义
2. 技术方案未明确确定
3. 验收标准未清晰定义
4. 用户未明确确认前进方向

### 与 6A 工作流整合
- Align 阶段 → Proposal + Specs
- Architect 阶段 → Design
- Atomize 阶段 → Tasks
- Approve 阶段
- Automate 阶段
- Assess 阶段

## 3. 智能体类型（L2-B001.3）

### 3.1 MCP服务智能体
- **功能**：提供各种服务能力，如文件系统、GitHub等
- **配置**：`.trae/mcp/` 目录下的服务配置
- **注册**：通过 `mcp.json` 或 `config.json` 配置文件注册

### 3.2 插件智能体
- **功能**：扩展系统功能的插件
- **配置**：`.trae/plugins/` 目录下的插件配置
- **注册**：自动扫描插件目录并注册

### 3.3 技能智能体
- **功能**：提供特定技能的智能体
- **配置**：`.trae/skills/` 目录下的技能配置
- **注册**：通过技能文件自动注册

## 4. 智能体管理（L2-B001.4）

### 4.1 智能体注册
1. **MCP服务**：在 `.trae/mcp/` 目录下创建服务配置文件
2. **插件**：将插件放入 `.trae/plugins/` 目录
3. **技能**：将技能文件放入 `.trae/skills/` 目录

### 4.2 智能体执行
- **通过工作流**：使用 `.trae/workflows/` 目录下的工作流执行智能体任务
- **通过API**：调用智能体API执行任务
- **通过CLI**：使用命令行工具执行智能体任务

## 12. 三层双向同步配置（L2-B001.12）

### 12.1 同步关系
根据 L1 层规范（L1-A001.6），本项目启用三层双向同步：

| 层级 | 路径 | 同步方向 |
|------|------|---------|
| L1 | `/agent.md` | ↔ L2 |
| L2 | `/.trae/agent.md` | ↔ L1, ↔ L3 |
| L3 | `/.trae/rules/*` | ↔ L2 |

### 12.2 同步触发条件
- **L1 → L2 → L3**：顶层规则变更时向下同步
- **L3 → L2 → L1**：底层规则变更时向上同步
- **L2 变更**：同时向 L1 和 L3 双向同步

### 12.3 冲突解决策略
- L1 与 L2 冲突：L1 为准，L2 自动修正
- L2 与 L3 冲突：L2 为准，L3 自动修正
- L1 与 L3 直接冲突：触发人工审核

### 12.4 同步配置
```yaml
sync:
  enabled: true
  auto_sync: true
  conflict_strategy: "l1优先"
  log_level: "info"
  sync_interval: 60
```

### 12.5 同步状态指示
- ✅ 已同步：三层规则完全一致
- ⚠️ 待同步：存在未同步的变更
- ❌ 冲突：检测到规则冲突，需人工处理

## 14. 蜂群模式与多智能体协作（L2-B001.14）

### 14.1 蜂群模式概述（L2-B001.14.1）

蜂群模式（Swarm Mode）是多智能体协作的去中心化模式，核心特征：

| 特征 | 说明 |
|------|------|
| 去中心化 | 无全局控制器，每个Agent独立决策 |
| 局部交互 | Agent只与邻居/相关方通信 |
| 自组织 | 宏观行为从微观交互中涌现 |
| 鲁棒性 | 单个Agent失败不导致系统崩溃 |

### 14.2 协作模式分类（L2-B001.14.2）

| 模式 | 结构 | 适用场景 |
|------|------|---------|
| 同级协作 | Agent间平等协作 | 分布式问题解决 |
| 层级协作 | 上下级指挥链 | 复杂任务分解 |
| 群峰模式 | 多个子团队协同 | 大规模系统工程 |
| 编排-工作者 | 中央编排+执行器 | 流水线处理 |

### 14.3 CrewAI最佳实践（L2-B001.14.3）

#### 14.3.1 角色定义
- 每个Agent有明确角色和职责
- 角色之间有清晰边界

| 流程 | 适用场景 |
|------|---------|

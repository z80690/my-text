## 3. 智能体类型（L2-B001.3）

### 3.1 MCP服务智能体
- **功能**：提供各种服务能力，如文件系统、GitHub等
- **配置**：`.trae/mcp/` 目录下的服务配置
- **注册**：通过 `mcp.json` 或 `config.json` 配置文件注册

### 3.2 插件智能体
- **功能**：扩展系统功能的插件
- **配置**：`.trae/plugins/` 目录下的插件配置
- **注册**：自动扫描插件目录并注册

### 3.3 技能智能体
- **功能**：提供特定技能的智能体
- **配置**：`.trae/skills/` 目录下的技能配置
- **注册**：通过技能文件自动注册

## 4. 智能体管理（L2-B001.4）

### 4.1 智能体注册
1. **MCP服务**：在 `.trae/mcp/` 目录下创建服务配置文件
2. **插件**：将插件放入 `.trae/plugins/` 目录
3. **技能**：将技能文件放入 `.trae/skills/` 目录

### 4.2 智能体执行
- **通过工作流**：使用 `.trae/workflows/` 目录下的工作流执行智能体任务
- **通过API**：调用智能体API执行任务
- **通过CLI**：使用命令行工具执行智能体任务

## 13. 三层双向同步配置（L2-B001.13）

### 13.1 同步关系（L2-B001.13.1）
根据 L1 层规范（L1-A001.6），本项目启用三层双向同步：

| 层级 | 路径 | 同步方向 |
|------|------|---------|
| L1 | `/agent.md` | ↔ L2 |
| L2 | `/.trae/agent.md` | ↔ L1, ↔ L3 |
| L3 | `/.trae/rules/*` | ↔ L2 |

### 13.2 同步触发条件（L2-B001.13.2）
- **L1 → L2 → L3**：顶层规则变更时向下同步
- **L3 → L2 → L1**：底层规则变更时向上同步
- **L2 变更**：同时向 L1 和 L3 双向同步

### 13.3 冲突解决策略（L2-B001.13.3）
- L1 与 L2 冲突：L1 为准，L2 自动修正
- L2 与 L3 冲突：L2 为准，L3 自动修正
- L1 与 L3 直接冲突：触发人工审核

### 13.4 同步配置（L2-B001.13.4）
```yaml
sync:
  enabled: true
  auto_sync: true
  conflict_strategy: "l1优先"
  log_level: "info"
  sync_interval: 60
```

### 13.5 同步状态指示（L2-B001.13.5）
- ✅ 已同步：三层规则完全一致
- ⚠️ 待同步：存在未同步的变更
- ❌ 冲突：检测到规则冲突，需人工处理

## 15. 蜂群模式与多智能体协作（L2-B001.15）

### 15.1 蜂群模式概述（L2-B001.15.1）

蜂群模式（Swarm Mode）是多智能体协作的去中心化模式，核心特征：

| 特征 | 说明 |
|------|------|
| 去中心化 | 无全局控制器，每个Agent独立决策 |
| 局部交互 | Agent只与邻居/相关方通信 |
| 自组织 | 宏观行为从微观交互中涌现 |
| 鲁棒性 | 单个Agent失败不导致系统崩溃 |

### 15.2 协作模式分类（L2-B001.15.2）

| 模式 | 结构 | 适用场景 |
|------|------|---------|
| 同级协作 | Agent间平等协作 | 分布式问题解决 |
| 层级协作 | 上下级指挥链 | 复杂任务分解 |
| 群峰模式 | 多个子团队协同 | 大规模系统工程 |
| 编排-工作者 | 中央编排+执行器 | 流水线处理 |

### 15.3 CrewAI最佳实践（L2-B001.15.3）

#### 15.3.1 角色定义（L2-B001.15.3.1）
- 每个Agent有明确角色和职责
- 角色之间有清晰边界
- 避免角色重叠

#### 15.3.2 任务分配（L2-B001.15.3.2）

| 能力匹配 | 任务匹配Agent核心能力 |
| 负载均衡 | 避免单一Agent过载 |

#### 15.3.3 流程类型（L2-B001.15.3.3）
| 流程 | 适用场景 |
|------|---------|
| sequential | 顺序依赖任务 |
| hierarchical | 层级指挥结构 |
| consensual | 协商一致决策 |

### 15.4 协作质量保障（L2-B001.15.4）

### 17.3 记忆类型定义（L2-B001.17.3）

| 类型 | 用途 | 隐私作用域 | 触发时机 |
|------|------|-----------|---------|
| `user` | 用户角色定义 | 用户私有 | 用户自我介绍时 |
| `feedback` | 用户反馈 | 用户私有 | 用户纠正或认可时 |
| `project` | 项目上下文 | 项目共享 | 检测到项目特定规则时 |
| `reference` | 外部参考 | 项目共享 | 用户提供参考链接时 |

### 17.4 记忆格式规范（L2-B001.17.4）

每条记忆必须包含：
| 字段 | 说明 | 示例 |
|------|------|------|
| **规则本身** | 清晰陈述事实或规则 | "不要在回复末尾加总结" |
| **Why** | 决策的理由 | "用户明确要求过" |
| **How to apply** | 触发该记忆的场景 | "所有回复结束时" |

### 17.5 记忆准入原则（L2-B001.17.5）

**核心原则**：只记录无法从代码和 Git 历史中推导出的信息。

| 不记录 | 原因 |
|--------|------|
| 代码模式、文件结构 | 可通过 grep 获取 |
| Git 历史 | git log 是权威来源 |
| CLAUDE.md 已存在内容 | 避免重复 |
| 临时任务状态 | 会话级信息无需持久化 |

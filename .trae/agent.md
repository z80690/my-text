# 智能体配置

## 1. 智能体目录结构

```
.trae/
├── mcp/                 # MCP服务配置
├── plugins/             # 插件目录
├── skills/              # 技能目录
├── rules/               # 规则和规范
├── sandbox/             # 沙箱配置
└── workflows/           # 工作流配置
```

## 2. 智能体类型

### 2.1 MCP服务智能体
- **功能**：提供各种服务能力，如文件系统、GitHub等
- **配置**：`.trae/mcp/` 目录下的服务配置
- **注册**：通过 `mcp.json` 或 `config.json` 配置文件注册

### 2.2 插件智能体
- **功能**：扩展系统功能的插件
- **配置**：`.trae/plugins/` 目录下的插件配置
- **注册**：自动扫描插件目录并注册

### 2.3 技能智能体
- **功能**：提供特定技能的智能体
- **配置**：`.trae/skills/` 目录下的技能配置
- **注册**：通过技能文件自动注册

## 3. 智能体管理

### 3.1 智能体注册
1. **MCP服务**：在 `.trae/mcp/` 目录下创建服务配置文件
2. **插件**：将插件放入 `.trae/plugins/` 目录
3. **技能**：将技能文件放入 `.trae/skills/` 目录

### 3.2 智能体执行
- **通过工作流**：使用 `.trae/workflows/` 目录下的工作流执行智能体任务
- **通过API**：调用智能体API执行任务
- **通过CLI**：使用命令行工具执行智能体任务

## 4. 智能体配置示例

### 4.1 MCP服务配置示例
```json
{
  "servers": {
    "filesystem": {
      "type": "local",
      "command": ["npx", "@modelcontextprotocol/server-filesystem"],
      "enabled": true,
      "timeout": 30000
    },
    "github": {
      "type": "git_hub",
      "enabled": true,
      "timeout": 30000
    }
  }
}
```

### 4.2 技能配置示例
```json
{
  "name": "code_analyzer",
  "description": "代码分析智能体",
  "tools": ["file_read", "code_execution"],
  "parameters": {
    "model": "gpt-4",
    "temperature": 0.7
  }
}
```

## 11. 博弈调度核心模块

### 11.1 模块目标
本模块扩展调度能力，使其能处理需要多视角碰撞或深度优化的复杂任务。通过两种博弈工作流，激发智能体间协作与制衡，产出更全面、更高质量的解决方案。

### 11.2 博弈模式定义与触发机制
在"任务分析"步骤中，必须根据以下规则识别并触发对应模式：

| 模式 | 触发信号 | 适用场景 | 工作流 |
|------|----------|----------|--------|
| **模式一：辩论模式** | 辩论、对比、优缺点、风险评估、多角度分析、正反两面 | 需要权衡利弊、评估风险、或从多个对立视角审视的问题 | 视角A → 视角B → 裁决 |
| **模式二：降维打击** | 优化、改进、重构、提升质量、润色、无懈可击 | 对已有方案或初稿进行深度优化和质量跃升 | 生成 → 挑剔 → 融合 |

### 11.3 博弈工作流详细规则

#### 模式一执行流程（串行模拟并行）
1. **识别对立视角**：根据任务，确定需要哪两种对立视角（例如："创新探索"对"风险控制"）
2. **顺序调用与信息传递**：
   - 首先，调用"视角A智能体"，给予完整原始任务
   - 将其完整输出，标记为"视角A结论"
   - 接着，调用"视角B智能体"。指令必须明确："请针对以下观点，从你的视角提出反对意见或不同看法：[此处插入视角A结论]"
   - 将其完整输出，标记为"视角B结论"
3. **最终裁决**：基于"视角A结论"和"视角B结论"，进行综合分析，输出一份包含"共识"、"核心分歧"和"综合建议"的最终报告

#### 模式二执行流程（流水线博弈）
1. **生成阶段**：调用最擅长"创意生成"的智能体，产出初始方案。标记此输出为"方案初稿"
2. **挑剔阶段**：调用最擅长"逻辑审查与挑错"的智能体。指令必须为："请对以下方案进行严格挑剔，找出所有潜在问题和漏洞：[此处插入方案初稿]"。标记此输出为"审阅意见"
3. **融合阶段**：调用最擅长"综合与重写"的智能体。指令必须为："请根据以下初稿和审阅意见，输出一份融合优化的最终方案。初稿：[方案初稿]。意见：[审阅意见]"
4. **最终交付**：将融合阶段智能体的输出，作为本次任务的最终结果交付

### 11.4 输出格式规范
当触发博弈模式时，整个响应必须严格遵循以下结构：

**第一步：分析结论**
明确说明识别的任务模式是"模式一"还是"模式二"，并简述原因。

**第二步：工作流规划**
详细说明将按照哪种工作流的哪个步骤执行，第一步调用谁，任务指令是什么，预期产出如何传递给下一步。

**第三步：下一步动作**
清晰告知即将执行工作流的第一步（例如："现在，我将开始执行【模式一】的第一步，调用视角A智能体"）

### 11.5 模块集成与优先级说明
- 博弈模块的触发判断，**优先于**常规的任务类型匹配
- 一旦触发任一博弈模式，则全程按本模块规则执行，不再走常规单智能体路由
- 本模块是多智能体协作能力在质量与深度上的专项增强

### 11.6 触发关键词速查表

| 模式 | 关键词（任一触发） |
|------|-------------------|
| **模式一** | 辩论、对比、优缺点、风险评估、多角度分析、正反两面 |
| **模式二** | 优化、改进、重构、提升质量、润色、无懈可击 |

### 11.7 智能体视角库

| 对立视角A | 对立视角B | 适用任务类型 |
|-----------|-----------|--------------|
| 创新探索 | 风险控制 | 技术选型、架构决策 |
| 乐观预期 | 悲观预期 | 项目评估、进度规划 |
| 用户中心 | 技术可行 | 产品设计、功能规划 |
| 短期效益 | 长期价值 | 技术债清理、架构升级 |
| 开发效率 | 代码质量 | 重构决策、技术实践 |

# 规则体系运作说明

**版本**: v1.0 | **更新日期**: 2026-05-27 | **适用项目**: my-text

---

## 一、体系概述

本项目采用 **三层规则体系 + 工作流引擎 + 记忆系统** 的架构模式，确保任务执行的质量、安全和可追溯性。

### 1.1 核心架构

```
┌─────────────────────────────────────────────────────────────────┐
│                        规则体系架构                            │
└─────────────────────────────────────────────────────────────────┘

    ┌─────────────────┐
    │   用户任务输入   │
    └────────┬────────┘
             │
             ↓
┌─────────────────────────────────────────────────────────────────┐
│                      规则引擎 (Rule Engine)                    │
│   ├─ 规则解析 → 优先级计算 → 冲突检测 → 流程协调 → 模块调度    │
│   └─ 不直接执行任务，协调工作流引擎执行                         │
└──────────────────────────┬──────────────────────────────────────┘
                           │
                           ↓
┌─────────────────────────────────────────────────────────────────┐
│                     工作流引擎 (6A流程)                        │
│   Align → Architect → Atomize → Approve → Automate → Assess   │
└──────────────────────────┬──────────────────────────────────────┘
                           │
                           ↓
┌─────────────────────────────────────────────────────────────────┐
│                    动态模块加载层                              │
│   质量保障 | 安全审计 | 测试指南 | 动态身份 | 任务拆分 | ...    │
└──────────────────────────┬──────────────────────────────────────┘
                           │
                           ↓
┌─────────────────────────────────────────────────────────────────┐
│                    质量门禁检查                                │
│   四维质量评分 → 综合评分 ≥75分 → 验收通过                      │
└──────────────────────────┬──────────────────────────────────────┘
                           │
                           ↓
┌─────────────────────────────────────────────────────────────────┐
│                      记忆系统                                  │
│   对话摘要 | 关键决策 | 任务状态 | 成功模式 | 失败教训          │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| **质量至上** | 所有规则的核心目标是保障任务交付质量 |
| **安全第一** | 零容忍硬编码、风险操作需审批 |
| **规则分层** | L1宪法 > L2基本法 > L3执行细则 |
| **流程强制** | 所有任务必须经过6A工作流处理 |
| **独立监督** | 质量评审员全程监督，拥有强制终止权 |
| **记忆持久** | 规则存储在文件系统，跨会话自动沿用 |

```
L1 (宪法层) - 根本原则，不可逾越
    ↓
L2 (基本法层) - 协作规范，必须遵守
```

## 三、规则引擎工作机制

### 3.1 执行流程

```
用户任务 → 规则解析 → 优先级计算 → 冲突检测 → 流程协调 → 模块调度 → 结果聚合 → 输出
```

### 3.2 规则优先级体系

| 优先级 | 类型 | 说明 |
|--------|------|------|
| **P0** | 用户输入 | 最高优先级，用户直接指令 |
| **P1** | 安全规则 | 零容忍硬编码、风险审批 |
| **P2** | 质量规则 | 四维质量模型、质量门禁 |
| **P3** | 流程规则 | 6A工作流、任务拆分 |
| **P4** | 效率规则 | Token优化、缓存策略 |

### 3.3 规则冲突解决

1. **同层级冲突**: 按优先级排序解决
2. **跨层级冲突**: 上层规则优先（L1 > L2 > L3）
3. **无法解决**: 质量评审员直接裁决

## 四、6A工作流详解

### 4.1 流程全图

```
Align（需求对齐）→ Architect（方案设计）→ Atomize（任务拆解）
       ↓                   ↓                   ↓
    需求确认            方案对比            任务清单
       ↓                   ↓                   ↓
Approve（审批确认）→ Automate（任务执行）→ Assess（评估验收）
       ↓                   ↓                   ↓
    审批通过            调度执行            质量评分
```

### 4.2 各阶段职责

| 阶段 | 职责 | 输出 | 检查点 |
|------|------|------|--------|
| **Align** | 需求对齐，澄清边界 | 需求确认文档 | 需求是否清晰？范围是否明确？ |
| **Architect** | 方案设计，2种方案对比 | 技术方案文档 | 方案是否合理？风险是否评估？ |
| **Atomize** | 任务拆解为原子任务 | 任务清单 | 拆分是否到位？粒度是否合适？ |
| **Approve** | 关键节点获得用户确认 | 审批通过/拒绝 | 审批是否到位？风险是否可控？ |
| **Automate** | 调度各功能模块执行 | 执行结果 | 规则是否遵循？质量是否达标？ |
| **Assess** | 质量门禁检查 | 质量评分 | 评分是否达标？门禁是否通过？ |

### 4.3 监督机制

每个阶段完成后，质量评审员进行独立检查：
- ✅ 通过：继续下一阶段
- ⚠️ 警告：需注意，可继续
- ❌ 否决：必须修复后才能继续

## 六、质量保障体系

### 6.1 四维质量模型

| 维度 | 定义 | 检查重点 |
|------|------|----------|
| **准确性** | 代码、配置、命令准确可执行 | 语法正确、逻辑正确、无运行时错误 |
| **安全性** | 代码和配置符合安全标准 | 无硬编码、依赖安全、权限合规 |
| **可维护性** | 代码易于理解和维护 | 命名规范、注释清晰、结构合理 |
| **可测试性** | 代码具备良好的测试基础 | 测试覆盖、接口稳定、解耦设计 |

### 6.2 质量度量标准

| 指标 | 达标标准 | 检测工具 |
|------|----------|----------|
| 单元测试覆盖率 | ≥70% | pytest, jest, coverage.py |
| 静态检查告警 | 0 critical | flake8, eslint, mypy |
| 安全审计问题 | 0 high | npm audit, pip-audit |
| 代码复杂度 | Cyclomatic < 15 | radon, eslint-complexity |

### 6.3 质量门禁规则

| 分数区间 | 门禁结果 | 处理方式 |
|----------|----------|----------|
| 90-100分 | ✅ 直接通过 | 可直接合并/部署 |
| 75-89分 | ⚠️ 条件通过 | 需要代码审查后通过 |
| 60-74分 | ⏳ 修复后重评 | 修复问题后重新评估 |
| 0-59分 | ❌ 拒绝通过 | 必须修复后重新提交 |

## 七、记忆系统

### 7.1 记忆结构

```
┌─────────────────┐    ┌─────────────────┐    ┌─────────────────┐
│  短期记忆层     │    │   中期记忆层    │    │   长期记忆层    │
│  (对话历史)     │    │  (关键决策)     │    │  (规则文件)     │
│  TTL: 会话周期  │    │  TTL: 任务周期  │    │  TTL: 持久      │
└─────────────────┘    └─────────────────┘    └─────────────────┘
```

### 7.2 对话压缩策略

| 对话轮数 | 压缩级别 | 策略 |
|----------|---------|------|
| < 20 | 无 | 保留全部对话 |
| 20-50 | 中等 | 最新10轮完整，其余转为摘要 |
| > 50 | 深度 | 最新10轮完整，40-10轮关键摘要，前40轮完整摘要 |

### 7.3 记忆内容分类

| 类型 | 内容 | 保留周期 |
|------|------|---------|
| 关键决策 | 技术选型、需求确认、规则变更 | 项目/持久 |
| 任务状态 | 6A阶段、进度、阻塞问题 | 实时 |
| 约束条件 | 安全约束、质量约束、环境约束 | 持久/项目 |
| 成功模式 | 成功的任务执行方法 | 持久 |
| 失败教训 | 失败原因和解决方案 | 持久 |

### 8.1 智能体类型

| 身份 | 职责 | 触发条件 |
|------|------|---------|
| 需求分析师 | 理解业务需求、澄清边界 | 需求不明确时 |

- **自动切换**: 根据任务阶段自动切换身份
- **上下文感知**: 根据子任务类型加载对应身份
- **多身份协作**: 复杂任务可同时激活多个身份

## 十、临时规则添加

### 10.1 添加方式

用户可直接在对话中提出临时规则，系统验证后立即生效：

**示例**:
```
临时规则：本次任务跳过代码审查，直接合并
```

### 10.2 临时规则优先级

| 优先级 | 规则来源 | 说明 |
|--------|----------|------|
| P0 | 用户输入（临时规则） | 当前会话有效，优先级最高 |
| P1 | L1规则（持久） | 宪法原则 |
| P2 | L2规则（持久） | 基本法规范 |
| P3 | L3规则（持久） | 执行细则 |

### 10.3 生命周期

- **生效**: 验证通过后立即生效
- **作用域**: 当前会话内有效
- **失效**: 会话结束后自动失效
